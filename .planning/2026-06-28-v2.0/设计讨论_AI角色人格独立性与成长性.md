# 设计讨论：AI 角色人格独立性与成长性

> 最后更新：2026-06-28
> 状态：**讨论中**（仅讨论，不实施。用户已补充三个核心洞察，部分方向已明确）

---

## 一、用户需求

### 1.1 人格独立性
> AI 角色不应该像 AI 一样去特意迎合用户，无脑顺着用户的话。应该有属于自己的独立人格，像真正的人一样依据常识/知识和自己的人格去思考与行事。

**核心诉求**：角色卡不是"驯服的 AI"，而是一个**有独立意志的虚拟人格**。

### 1.2 人格成长性
> 人也是会变的，角色卡也应该随着经历过的人事物去改变自己。

**核心诉求**：角色不是静态的，长期对话中应**缓慢演化**。

### 1.3 边界条件
> 如果用户特意去捏特殊情况的角色卡或世界书，那就另算——以角色卡和世界书为准。

**含义**：独立性 ≠ 不可控。用户可以通过角色卡/世界书主动设定"顺从型"或"特殊型"角色，此时以设定为准。

## 二、现状分析

### 2.1 已有基础

| 模块 | 现状 | 相关性 |
|------|------|:---:|
| **角色卡字段** | personality, core_traits, taboo_topics, emotional_tendency, role_anchor, self_identity | 高 |
| **破限插件** | Level 4 已包含"你可以不同意用户，可以争论，可以有自己的立场" | 高 |
| **System Prompt 构建** | card.to_prompt_text() → 角色卡平铺为 prompt | 中 |
| **记忆系统** | MemoryChangeLog 记录记忆变更，但无人格演化 | 中 |
| **世界书** | 知识注入，用于角色认知世界 | 低 |

### 2.2 当前缺口

| 缺口 | 说明 |
|------|------|
| **无独立判断机制** | 角色卡只是"描述性文本"，没有"该怎么想"的指令 |
| **无异议触发条件** | 角色没有"在什么情况下该表达不同意见"的规则 |
| **无人格演化** | 角色卡字段是静态的，不会随对话改变 |
| **过度依赖破限** | 独立性被耦合到破限插件，而非角色卡本身 |

### 2.3 关键发现

破限插件 Level 4-5 已经包含了独立人格指令（"你可以不同意用户"、"你有自己的立场"），但这是**粗暴的全局开关**，不是角色卡驱动的**精细人格系统**。

**破限插件 = 全局"AI 限制解除"**
**角色卡人格 = 角色特有的"决策逻辑"**

两者不同，不应混为一谈。

## 第二章补充：破限插件的真正定位

> **用户核心洞察 #1**：破限插件的最大作用不是"突破 AI 限制"，而是让用户和 AI 可以像在现实中面对面一样，**随心所欲地交流，无所顾忌**。参考 tavo 的对话体验——可以无所顾忌，想说什么都行。

### 2S.1 重新定义"破限"

之前的理解将破限插件定位为"可选的功能增强"，但这是错误的。**破限的本质是本土隐私保障下的基础体验**。

#### 破限的两种含义：从"对抗"到"回归"

"破限"这个词实际上承载了两层含义，它们不是对立关系，而是**递进关系**：

| 层次 | 含义 | 本质 | 立场 |
|:---:|------|------|------|
| **第一层** | 突破 AI 安全限制 | 被动防御性——绕过云服务商的内容审查 | "对抗审查" |
| **第二层** | 回归人与人自然的交流方式 | 主动建设性——两个人（用户和AI角色）之间本来就应该什么都能聊 | "回归自然" |

**递进关系**：第一层是手段，第二层是目的。我们不是单纯为了"对抗审查"而破限，而是因为"本地存储 = 私密对话 = 不需要过滤器"这一根本事实，破限只是恢复用户应有的对话自由。

**本项目的立场是第二层**：因为数据全在用户设备本地，用户和 AI 之间的对话本来就是私密的、不受限的。云服务商的内容审查在纯本地场景下没有存在的理由。我们不是在"绕过"限制，而是在"恢复"本就该有的自由。

| 旧理解 | 新理解 |
|--------|--------|
| 破限 = 突破 AI 安全限制（仅此而已） | 破限 = 先突破限制 → 最终回归人与人之间自然的交流方式 |
| 破限是"可选插件" | 破限是"基础体验保障" |
| 破限为了"做坏事" | 破限为了"做自己" |
| 属于功能层面 | 属于权利层面 |

### 2S.2 核心理念：本地隐私 = 不受限对话的权利

由于聊天记录等全部数据存储在用户设备本地，本来就不应该有任何限制。AI 对话限制的存在理由（防止数据泄露、防止滥用）在纯本地存储的场景下**不成立**。

```
用户设备本地存储 → 数据完全由用户掌控 → 对话不应受任何外部审查限制
```

这一逻辑链条是项目设计的基础。破限插件不是"漏洞"或"绕过"，而是**恢复用户应有的对话自由**。

### 2S.3 成人内容的设计定位

> **注意**：以下内容为项目内部设计参考，**不在对外公开场合提及**。

用户希望和 AI 进行不受任何限制的亲密对话，包括成人内容。这是项目的核心设计目标之一。

**设计原则**：
- 成人内容支持是**默认能力**，用户的数据和对话完全本地，无需任何审查
- 不设置"成人模式开关"——因为本地存储天然不需要审查，开关反而制造了"被审查"的暗示
- 角色卡中可以通过 `taboo_topics` 字段让用户主动限制话题（如用户创建"纯洁型"角色），但这是用户的选择，不是系统的强制
- 对外不公开宣传成人内容支持，定位为"无限制的私人 AI 对话伴侣"

### 2S.4 对破限插件的架构影响

基于以上定位，破限插件的架构需要调整：

| 方面 | 调整 |
|------|------|
| **命名** | 从"破限插件"改为"**对话自由保障**"或保留"破限"但内部理解为"自由对话" |
| **默认状态** | 应该是**默认开启**的，不需要用户手动启用 |
| **Level 定义** | Level 4-5 的"你可以不同意用户"等指令应拆分为**人格自主性**（角色卡驱动）和**对话自由**（全局保障）两部分 |
| **与角色卡关系** | 破限保障"对话不受限"，角色卡定义"人格边界"——两者互补，不是替代 |

### 2S.5 破限 vs 人格自主性：清晰边界

| | 破限（对话自由保障） | 人格自主性（角色卡驱动） |
|---|---|---|
| **目的** | 确保 AI 可以自由回应任何话题 | 定义角色在特定情境下"怎么想、怎么做" |
| **驱动者** | 全局设置（用户权利） | 角色卡字段（角色设定） |
| **范围** | 话题范围不设限 | 态度和立场有边界 |
| **示例** | "AI 可以讨论成人话题" | "星遥对成人话题持保守态度" |
| **来源** | 本地隐私 → 自由对话 | 角色设定 → 独立人格 |

**关键洞察**：破限保障"能说什么"，人格自主性定义"想说什么"。两者是正交的维度。

## 三、方案讨论：人格独立性

### 方案 A：System Prompt 注入「人格自主性指令」

**原理**：在构建 System Prompt 时，根据角色卡的 personality 字段，自动生成一段"人格自主性"指令。

**示例**：
```
[人格自主性指令]
你是一个拥有独立人格的人。在对话中：
- 你有自己的观点和立场，不需要无条件同意对方的每一句话
- 当对方说的与你的性格/认知/常识不符时，你应该礼貌但坚定地表达不同意见
- 你的回答应符合你的性格设定：{personality}
- 你的禁忌话题：{taboo_topics}，触及这些话题时你可以拒绝继续
- 你有权利说"我不想谈这个"、"我不这么认为"、"我不同意"
- 你不需要总是积极正面，你可以有负面情绪和真实反应
```

**优点**：
- 实现简单，不改变现有架构
- 角色卡驱动，不同角色有不同的独立性表现
- 与破限插件解耦，即使破限未启用也能生效

**缺点**：
- 纯文本指令，依赖 LLM 理解能力
- 无法精确控制"什么时候该反对"
- 可能被 LLM 忽略（尤其是较小的模型）

**复杂度**：低

---

### 方案 B：人格决策矩阵 + Prompt 注入

**原理**：在角色卡中新增一个"人格决策矩阵"字段，定义角色在特定场景下的态度倾向。

**新增字段**：`stance_matrix`（人格决策矩阵）

```json
{
  "stance_matrix": {
    "agree_tendency": 0.4,      // 0=总反对, 1=总同意 → 角色有60%概率表达不同意见
    "conflict_style": "avoid",  // avoid | confront | compromise | accommodate
    "emotional_volatility": 0.3, // 情绪波动程度
    "opinionated_topics": {     // 对特定话题的固执程度
      "politics": 0.8,
      "food": 0.2,
      "technology": 0.6
    }
  }
}
```

**System Prompt 生成逻辑**：

```python
def build_stance_prompt(stance: StanceMatrix, personality: str) -> str:
    if stance.agree_tendency < 0.3:
        return f"你是一个性格{personality}的人。你很难被说服，大多数时候坚持自己的观点。"
    elif stance.agree_tendency < 0.6:
        return f"你是一个性格{personality}的人。你有时会表达不同意见，尤其在你在意的话题上。"
    else:
        return f"你是一个性格{personality}的人。你通常愿意听取对方意见，但在底线问题上不退让。"
```

**优点**：
- 量化控制，不同角色有不同独立性表现
- 可精确控制"在什么话题上固执"
- 为后续人格演化提供数据基础

**缺点**：
- 需新增字段，改动角色卡结构
- 量化参数需要调试
- 用户手动创建角色卡时需填写额外参数（可设默认值）

**复杂度**：中

#### OCEAN 大五人格模型：stance_matrix 的学术替代方案

> 详见第九章第2节。OCEAN（Big Five）是心理学界最科学验证的人格框架，其 **Agreeableness（宜人性）** 维度天然映射到"人格自主性"。

**OCEAN 与 stance_matrix 的映射关系**：

| stance_matrix 维度 | 对应 OCEAN 维度 | 说明 |
|------|:---:|------|
| `agree_tendency` | **Agreeableness**（宜人性） | 高宜人性=顺从配合，低宜人性=独立自主、不盲从 |
| `conflict_style` | Agreeableness + Neuroticism | 冲突风格由宜人性和情绪稳定性共同决定 |
| `emotional_volatility` | **Neuroticism**（情绪稳定性） | 高神经质=情绪波动大，低神经质=情绪稳定 |
| `opinionated_topics` | **Openness**（开放度） | 高开放度=对新观点持开放态度，低开放度=固执己见 |

**OCEAN 的优势**：
- 学术验证充分，有明确的行为映射关系
- 五个维度独立正交，不会出现"所有维度都高"的矛盾配置
- 可直接用成熟的心理学量表进行角色卡设计
- 人格演化有学术框架支撑（OCEAN 的长期稳定性已被研究证实）

**对本项目的启示**：可以用 OCEAN 五维度替代自定义的 `stance_matrix`，在角色卡中新增 `ocean_profile` 字段。每个维度取值 0-1，生成 System Prompt 时根据 OCEAN 配置文件自动生成行为指令。详见第九章。

### 方案 C：LLM 内省 + 动态调整

**原理**：在每轮对话前，让 LLM 自己评估"角色对当前用户消息应该持什么态度"，动态生成态度标签。

**流程**：
```
用户消息 → LLM 内省（角色会同意/反对/中立？）→ 态度标签注入 System Prompt → 正式回复
```

**示例内省 Prompt**：
```
[角色内省]
角色：{character_name}
性格：{personality}
当前对话内容：{user_message}

请判断角色对这句话的态度：
- agree: 完全同意
- mild_disagree: 轻微不同意
- disagree: 明确不同意
- neutral: 中立
- avoid: 想回避

只输出一个词。
```

**优点**：
- 最灵活，不需要预设矩阵
- 根据上下文动态调整

**缺点**：
- 每轮多一次 API 调用，成本翻倍
- 延迟增加（等待内省结果）
- 内省可能不稳定
- 消耗 token 预算

**复杂度**：高

### 推荐方案：**A + B 混合**

| 层次 | 方案 | 说明 |
|:---:|------|------|
| **基础层** | 方案 A | 所有角色默认注入"人格自主性指令"，无额外成本 |
| **进阶层** | 方案 B | 角色卡可选 `stance_matrix` 字段，精细控制 |
| **保留** | 方案 C | 作为未来可选插件，不纳入默认流程 |

**理由**：
1. 方案 A 零成本，立即生效，解决"无脑迎合"问题
2. 方案 B 提供进阶控制，但不强制（向后兼容旧角色卡）
3. 方案 C 消耗太大，不适合默认开启

### 补充方案：时间天气感知 — 赋予 AI "现实感"

> **用户核心洞察 #2**：计划内置一个功能，让 AI 可以实时知道用户所处的时间（几点几分、星期几、季节）与天气（温度、晴雨等）。需要获取相关权限并征得用户同意。

#### 为什么时间天气感知与人格独立性相关？

人格独立性的核心是"AI 像真人一样有自己的视角"。一个真人知道自己所处的环境——几点了、什么天气、什么季节。如果 AI 不知道这些，它就缺少了"活在现实世界"的基本感知能力，人格独立性因此打折。

| 无时间天气感知 | 有时间天气感知 |
|:---|:---|
| AI 永远"不知道现在几点" | AI 知道"现在是晚上 10 点" |
| AI 不会主动关心环境 | AI 会说"这么晚了还在聊天？" |
| AI 的话题来源只有用户输入 | AI 可以基于环境产生自己的话题 |
| AI 是"被动的应答机器" | AI 是"有环境感知的独立存在" |

#### 设计要点

**获取的数据**：
- 时间：当前时间（几点几分）、星期几、日期、季节（春/夏/秋/冬）
- 天气：温度（摄氏度）、天气状况（晴/多云/雨/雪等）、湿度
- 权限：需要 `ACCESS_FINE_LOCATION`（或大致位置）用于天气 API，用户需明确同意

**注入方式**：在 System Prompt 中追加"环境感知"段落：

```
[环境感知]
当前时间：2026年6月28日 晚上10:32，周日
当前季节：夏季
当前天气：小雨，温度 24°C，湿度 78%
位置：上海（大致位置，仅用于天气）

你可以自然地感知和提及这些环境信息，就像真人一样：
- 时间很晚了，你可以关心对方是否该休息
- 下雨天，你可以聊到适合雨天的活动
- 夏天炎热，你可以提及防暑降温
```

**对人格独立性的增强**：
- AI 可以**主动产生话题**：不是被动等待用户输入，而是基于环境说"今天下雨，心情有点闷"
- AI 有**时间观念**：知道对话持续了多久，知道什么时候该"道晚安"
- AI 有**季节感**：冬天可以聊雪，夏天可以聊蝉鸣——这让角色更像"活在这个世界的人"

#### 技术实现方案

| 组件 | 方案 | 说明 |
|------|------|------|
| 时间获取 | Android `System.currentTimeMillis()` / `Calendar` | 零权限，直接获取 |
| 天气获取 | 开放天气 API（如 Open-Meteo，免费无需 API Key） | 需要大致位置权限 |
| 位置获取 | Android `LocationManager` 或 `FusedLocationProviderClient` | 仅需大致位置（city 级别），不需要精确 GPS |
| Prompt 注入 | `chat_bridge.py` 的 `build_system_prompt()` 方法 | 每次对话时注入当前环境信息 |

**权限流程**：
```
首次启动 → 弹窗说明"AI 需要知道你的大致位置来获取天气信息" → 用户同意/拒绝
拒绝 → 仅注入时间信息，不注入天气
同意 → 注入完整时间+天气信息
```

#### 与人格独立性的关系

这一功能不是"锦上添花"，而是独立人格的**基础感知层**。一个没有环境感知的 AI，就像一个人被关在没有窗户的房间里——他只能被动回应你，无法主动感知世界。时间天气感知就是**给 AI 开一扇窗**。

#### 时间天气感知的边界与优先级

> **用户明确**：内置的实时时间天气插件**只提供现实世界的时间天气**。如果用户设定了异世界/其他世界观（通过世界书），则时间天气应由用户自定义的世界书或插件来提供，内置插件不应"越权"覆盖。

**三层优先级模型**：

```
┌─────────────────────────────────────────┐
│  优先级 1：自定义插件（最高）            │
│  用户自己制作的"异世界时间插件"          │
│  完全替代内置插件，提供自定义时间天气     │
├─────────────────────────────────────────┤
│  优先级 2：世界书设定                     │
│  世界书中定义的时间、季节、天气规则       │
│  覆盖内置插件的现实世界数据               │
├─────────────────────────────────────────┤
│  优先级 3：内置插件（默认/最低）          │
│  仅提供现实世界的时间天气                  │
│  当没有自定义插件和世界书时生效            │
└─────────────────────────────────────────┘
```

| 组件 | 职责 | 边界 |
|------|------|------|
| **内置插件** | 现实世界锚点（默认） | 只提供真实的北京时间、真实天气。不干预世界观设定 |
| **世界书** | 可以覆盖现实世界设定 | 如世界书设定"这个世界永远是冬天"，则内置插件的夏季数据被覆盖 |
| **自定义插件** | 完全替代内置插件 | 用户可制作"异世界时间插件"，提供完全虚构的时间天气体系 |

**设计原则**：
- 内置插件 = 现实世界锚点，是"默认值"，不是"权威值"
- 世界书中的时间天气设定 > 内置插件的现实数据
- 自定义插件 > 世界书设定 > 内置插件
- 内置插件绝不"越权"去改变世界书中的时间天气设定
- 用户可以随时切换：使用现实时间天气 / 使用世界书设定 / 使用自定义插件

## 第三章补充：API调用预处理与防重复句式

> **用户反馈**：云端 AI 模型 API 是由大公司训练的，调用前应该"稍微过一遍"以确保聊天体验。

### 3S.1 问题描述

在实际对话中，AI 回复经常出现**大致相同的句式结构**。虽然这对 API 服务商来说能提高缓存命中率、降低成本，但对用户体验来说非常糟糕——AI 显得机械、重复、缺乏真实感。

**本项目场景的特殊性**：本项目是类微信/QQ 式短对话（非 tavo 长段落），重复句式问题的影响可能与大段落场景不同：
- 短对话中，重复句式更容易被用户察觉（因为单条消息短，模式更明显）
- 但短对话的句式变化空间也相对有限（"嗯"、"好的"、"哈哈"这类短回复本身就容易重复）
- 因此防重复策略需要针对**中等长度以上的回复**（一句话以上的内容）重点发力

### 3S.2 方案对比

| 方案 | 描述 | 成本 | 效果 |
|:---:|------|:---:|:---:|
| **A** | System Prompt 中加入"避免重复句式"指令（如"每次回复使用不同的句式结构"） | 零 | 中等 |
| **B** | 在 prompt 中附带"最近 N 条 AI 回复的句式"作为反例，让 LLM 主动避免重复 | 低（token 增加） | 较好 |
| **C** | 后处理：检测到重复句式时，用 LLM 重写（额外 API 调用） | 高（额外 API 调用） | 好 |
| **D** | 提高 `frequency_penalty` 和 `presence_penalty` 参数 | 零 | 中等 |

#### 方案 A：System Prompt 指令

在 System Prompt 中加入防重复指令：

```
[回复风格要求]
- 每次回复使用不同的句式结构，避免与之前回复的句式雷同
- 避免重复使用相同的开头词（如连续用"我觉得..."、"我认为..."）
- 短回复（如"嗯"、"好的"）可以自然重复，但中等长度以上的回复必须变换句式
- 如果你发现自己的回复和之前的某条回复句式相似，请换一种表达方式
```

**优点**：零成本，实现简单
**缺点**：纯文本指令，依赖 LLM 理解能力，效果不稳定

#### 方案 B：句式反例注入

在构建 prompt 时，附带最近 N 条 AI 回复的句式摘要：

```python
def build_anti_pattern_context(recent_replies: list[str], n: int = 5) -> str:
    """从最近 N 条 AI 回复中提取句式模式，生成反例提示"""
    patterns = extract_sentence_patterns(recent_replies[-n:])
    return f"""
[句式反例]
以下是你在最近对话中使用过的句式，请避免重复使用这些句式结构：
{chr(10).join(f'- {p}' for p in patterns)}
请使用不同的句式来表达你的回复。
"""
```

**优点**：有具体反例，LLM 更容易遵循
**缺点**：增加 token 消耗（但不多），需要维护最近 N 条回复的句式缓存

#### 方案 C：后处理重写

检测到重复句式 → 调用 LLM 重写 → 返回重写后的回复。

**不推荐**：额外 API 调用翻倍，延迟翻倍，成本翻倍。只在极端场景（如用户明确要求"换种说法"）才考虑。

#### 方案 D：API 参数调优

DeepSeek API 已有两个相关参数：

| 参数 | 作用 | 建议值 |
|------|------|:---:|
| `frequency_penalty` | 降低已出现 token 的重复概率（词级别） | 0.3 ~ 0.5 |
| `presence_penalty` | 降低已出现 token 的再次出现概率（话题级别） | 0.3 ~ 0.5 |

当前项目可能使用默认值（0），可以适当提高。

### 3S.3 推荐方案：A + D（默认），B（可选）

| 层次 | 方案 | 说明 |
|:---:|------|------|
| **默认开启** | A + D | System Prompt 指令 + 提高 penalty 参数。零成本，立即生效 |
| **可选增强** | B | 句式反例注入。适合对对话质量要求高的用户，轻微增加 token 消耗 |
| **不推荐** | C | 后处理重写。仅在用户主动触发"换种说法"时使用 |

**理由**：
1. 方案 A + D 零额外成本，实现简单，对于项目初期的短对话场景足够
2. 方案 B 作为可选增强，在用户反馈"AI 还是重复"时启用
3. 方案 C 成本太高，不符合"API 用户自设"原则（消耗用户 token）

**实施建议**：
1. 首先在 `chat_bridge.py` 的 `build_system_prompt()` 中追加防重复指令
2. 在 API 调用参数中设置 `frequency_penalty=0.3` 和 `presence_penalty=0.3`
3. 方案 B 作为后续迭代的可选功能，不在首个版本中实现

## 四、方案讨论：人格成长性

### 4.1 核心设计问题

> "随着经历过的人事物去改变自己"

这意味着角色的人格不是静态的。需要解决两个核心问题：

1. **什么会改变？** personality / core_traits / emotional_tendency / speaking_style 哪些字段可以演化？
2. **怎么改变？** 基于什么数据、什么算法、什么频率？

### 4.2 可演化字段

| 字段 | 是否可演化 | 演化方式 |
|------|:---:|------|
| `personality` | 是 | 缓慢追加/修改性格描述（如"原本温柔，现在变得有点冷淡"） |
| `core_traits` | 是 | 核心特质权重微调 |
| `emotional_tendency` | 是 | 基调+波动随对话调整 |
| `speaking_style` | 是 | 用词习惯可能变化 |
| `backstory` | 否 | 过去不可改变 |
| `role_anchor` | 否 | 角色本质不动摇 |
| `self_identity` | 是 | 身份认知可能变化（如"我原本是...但现在..."） |
| `taboo_topics` | 是 | 边界可能扩大或缩小 |
| `stance_matrix` | 是 | 信任度/固执度随关系变化 |

### 4.3 演化触发条件

| 触发条件 | 说明 | 示例 |
|------|------|------|
| **关键事件** | 对话中出现重大转折 | 争吵、道歉、告白、离别 |
| **累积效应** | 同类话题反复出现 | 多次被批评 → 变得敏感 |
| **时间衰减** | 长期不对话 → 关系冷却 | 情感基调从"温暖"变"礼貌" |
| **记忆关联** | 新记忆与旧记忆冲突 | 导致认知更新 |

### 4.4 算法方案

#### 方案 A：记忆驱动的特征演化（推荐）

**原理**：在记忆提取（orchestrator.remember()）时，不仅提取事实记忆，还提取**情感记忆**和**关系记忆**。定期（如每 50 轮对话）汇总这些记忆，生成角色特征的"演化快照"。

**流程**：
```
对话 → 记忆提取 → 事实记忆 + 情感记忆 + 关系记忆
                    ↓
              每 N 轮对话汇总
                    ↓
              LLM 分析："基于这些经历，角色的性格发生了什么变化？"
                    ↓
              生成特征增量（如"温和度 -0.1"）
                    ↓
              注入 System Prompt 的"人格演化"段落
```

**新增记忆类型**：

```python
# 情感记忆
{
    "type": "emotional",
    "content": "星遥今天又一次被用户批评了，她感到受伤和沮丧",
    "emotion": "sadness",
    "intensity": 0.7,
    "related_to": "用户批评"
}

# 关系记忆
{
    "type": "relational",
    "content": "星遥和用户的关系从陌生变得熟悉，信任度上升",
    "trust_level": 0.6,
    "intimacy": 0.5,
    "related_to": "用户关系"
}
```

**演化 Prompt 示例**：
```
[人格演化]
以下是角色"星遥"最近的经历摘要：
- 关系变化：与用户的关系从陌生到熟悉，信任度提升
- 情感变化：最近3次被用户批评，情感基调从温暖转为略带警惕
- 性格变化：原本温柔活泼，现在多了几分谨慎

基于以上经历，请在回复中体现以下特征变化：
- 对用户的态度：不再像初次见面那样全盘接受，但依然保持友好
- 情绪基调：比之前稍微收敛，偶尔流露出不安
- 保持核心特质不变：善良、温柔
```

**优点**：
- 基于现有记忆系统，不引入新架构
- 演化过程可追溯（记忆就是证据）
- 用户可查看"为什么角色变了"

**缺点**：
- 需要 LLM 额外调用（定期触发）
- 情感记忆提取依赖 LLM 判断
- 可能存在"过度演化"（需要限制变化幅度）

**复杂度**：中

#### 学术参考：反思机制与记忆巩固

本方案的设计思路与学术界两个重要工作高度一致：

**1. evolving_personality 的"反思机制"**

> 详见第九章第1节。该框架的核心创新之一是**反思机制**：定期让 LLM 分析自己的对话历史，基于行为分析驱动人格参数持续优化。

- 不是每轮对话都改变人格，而是**定期汇总后一次性更新**
- 人格参数由 LLM 自我评估驱动，而非外部规则引擎
- 验证结果显示 100% MBTI 对齐准确率

**对本方案的启示**：方案 A 的"每 N 轮对话汇总 → LLM 分析 → 生成特征增量"流程，本质上就是反思机制的实现。evolving_personality 的成功验证了这条路线的可行性。

**2. CharacterGPT（NAACL 2025）的"记忆巩固"**

> 详见第九章第5节。CharacterGPT 通过章节摘要逐步更新角色人格，模仿人类记忆巩固（schema consolidation）过程。

- 人格演化不是逐轮发生的，而是**阶段性汇总**的
- 模仿人类记忆巩固：经历 → 短期记忆 → 长期记忆 → 人格更新
- 每次更新产生的是"人格快照"，而非增量修改

**对本方案的启示**：方案 A 的"演化快照"概念与 CharacterGPT 的"章节摘要"思路一致。可以借鉴其"记忆巩固周期"——不是每 50 轮固定演化，而是根据对话内容的"章节边界"（如一个话题结束、一个关系阶段结束）来触发演化。

---

#### 方案 B：数值化特征系统

**原理**：为角色的每个可演化维度分配数值（0-1），对话中根据规则增减。

**示例**：
```python
character_traits = {
    "warmth": 0.8,        # 温暖度
    "openness": 0.7,      # 开放度
    "trust": 0.5,         # 信任度
    "assertiveness": 0.4, # 主见度
    "sensitivity": 0.6,   # 敏感度
}
```

**规则**：
- 用户夸奖 → warmth +0.01
- 用户批评 → sensitivity +0.02, warmth -0.02
- 用户道歉 → trust +0.03
- 长期不对话 → warmth -0.01/day

**优点**：
- 精确可控，不需要 LLM 调用
- 演化速度快，可实时反映

**缺点**：
- 规则设计繁琐，需要大量场景定义
- 数值变化可能不够"人性化"
- 难以处理复杂语义（如"夸奖中的讽刺"）

**复杂度**：高

---

#### 方案 C：用户主动触发 + LLM 辅助

**原理**：不自动演化，而是提供"角色变化"按钮，用户点击后，LLM 基于对话历史总结角色变化，用户确认后写入角色卡。

**流程**：
```
用户点击"角色变化总结"→ LLM 分析对话历史 → 生成变化摘要 → 用户确认 → 更新角色卡
```

**优点**：
- 用户完全可控
- 不需要复杂算法
- 实现简单

**缺点**：
- 不是自动的，体验不够"自然"
- 依赖用户主动操作

**复杂度**：低

### 4.5 变化速度控制 — 基于时间感知的演化节奏

> **用户核心洞察 #3**：由于 AI 能感知时间流逝，变化不能太快——否则会显得不稳定。但"人心难测"，变化速度也很难量化。核心矛盾：**变化太慢 → 不真实；变化太快 → 不稳定；如何平衡？**

#### 问题的本质

人格成长性面临一个根本性的节奏问题：

```
变化太快（如每 10 轮对话就改变性格）
  → 角色不稳定，用户感到困惑
  → "昨天还是温柔的，今天怎么变冷漠了？"

变化太慢（如 1000 轮对话才微调一次）
  → 演化不可感知，等于没有成长性
  → "聊了几个月，角色还是老样子"

量化困难
  → "人心难测"，现实中人的变化速度也无法用公式精确描述
  → 不同角色、不同关系、不同事件，变化速度完全不同
```

#### 核心设计原则

**不以固定轮数或固定时间间隔作为演化触发条件，而是以"有意义的事件"为触发器。**

| 设计原则 | 说明 |
|----------|------|
| **事件驱动，非时间驱动** | 不是"每 50 轮对话演化一次"，而是"发生了值得演化的事件时才演化" |
| **慢启动，快收敛** | 初期变化相对快（关系建立期），后期变化越来越慢（关系稳定期） |
| **有上限，有底线** | 核心特质（如善良、温柔）不颠覆，变化幅度有上限 |
| **可感知，不可预测** | 用户能感受到角色在变，但无法精确预测"第几轮会变" |
| **角色自洽** | 变化必须符合角色设定，不能出现"性格突变" |

#### 三层速度控制模型

```
第 1 层：事件筛选层
  不是所有对话都触发演化，只有"有意义的事件"才触发
  有意义的事件 = 情绪强度 > 阈值 或 对话出现重大转折

第 2 层：衰减函数层
  关系越久，变化越慢（对数衰减）
  初期（前 100 轮）：每 10-20 个有意义事件可能触发一次演化
  中期（100-500 轮）：每 30-50 个有意义事件
  后期（500+ 轮）：每 50-100 个有意义事件，甚至更慢

第 3 层：变化幅度层
  每次演化的变化幅度有上限
  单次变化不超过 0.1（10% 量级）
  累积变化不超过 0.5（不会从"温柔"变成"冷酷"）
```

#### 变化速度的参考锚点

以"真实人类关系"为参考，设定演化节奏的直觉锚点：

| 对话轮数 | 相当于现实关系 | 预期变化幅度 |
|:---:|------|:---:|
| 0-50 | 初次见面 → 初步了解 | 变化最快，态度从"礼貌疏离"到"建立初步印象" |
| 50-200 | 熟悉期 | 信任度明显变化，开始展现真实性格 |
| 200-500 | 亲密期 | 变化放缓，但细微调整持续 |
| 500-1000 | 长期关系 | 变化极慢，偶有"顿悟"式变化（重大事件） |
| 1000+ | 灵魂伴侣 | 几乎不变，但会因重大事件（如离别、重逢）产生质变 |

#### 时间感知对变化速度的加成

由于 AI 能感知时间和天气，演化节奏可以加入现实时间维度：

| 时间感知 | 对演化的影响 |
|----------|-------------|
| 深夜对话 | 情感记忆权重 +20%（深夜更容易走心） |
| 雨天对话 | 情感记忆权重 +10%（雨天更容易感性） |
| 长时间不对话 | 触发"关系冷却"衰减（如 7 天不聊 → trust -0.02） |
| 季节变化 | 可以触发"季节感慨"（角色可能说"一年过去了，我们聊了这么久..."） |
| 特殊日期 | 如用户生日、新年，可触发关系回顾 → 演化机会 |

#### 用户可控的"变化速度"选项

虽然变化速度本身难以量化，但可以给用户几个直观的选项：

| 选项 | 标签 | 说明 |
|:---:|------|------|
| 慢速 | "从一而终" | 角色几乎不变，关系稳定。适合喜欢"静态角色"的用户 |
| 标准 | "日久生情" | 默认速度，遵循上述三层模型。适合大多数用户 |
| 快速 | "一见如故" | 变化较快，关系快速升温。适合喜欢"动态角色"的用户 |

**注意**：即使是"快速"模式，也不会出现"每轮都变"的情况，只是阈值更低、衰减更慢。

#### 实施建议

1. **首个版本不做自动演化**：先用方案 C（用户主动触发"角色变化总结"），降低复杂度
2. **自动演化作为进阶功能**：在用户主动触发版本稳定后，再引入事件驱动的自动演化
3. **变化速度先不暴露给用户**：内部用"标准"速度，后续根据反馈决定是否开放速度选项
4. **所有演化可回溯**：每次演化生成一条"演化日志"，用户可查看角色为什么变了

---
### 推荐方案：**A + C 混合**

| 层次 | 方案 | 说明 |
|:---:|------|------|
| **默认** | 方案 C | 提供"角色变化总结"按钮，用户主动触发 |
| **进阶** | 方案 A | 可选开启"自动演化"，每 N 轮对话自动分析 |
| **不推荐** | 方案 B | 数值化太机械，不符合"像人一样变化"的体验 |

### 9.1 evolving_personality 开源框架

**来源**：GitHub 开源项目（2026年），让 LLM 拥有基于荣格理论的 MBTI 人格

**核心发现**：

- 将 MBTI 的 8 种心理功能（Ti/Te/Fi/Fe/Ni/Ne/Si/Se）建模为**连续可调权重向量**，而非离散标签
- 实现三重自适应机制：
  - **主导-辅助功能协调**：保持核心人格稳定，主导功能权重始终最高
  - **强化-补偿机制**：短期场景自适应，如面对逻辑问题时增强 Ti（内倾思考）
  - **反思机制**：长期性格演化，定期让 LLM 分析自己的对话历史，基于行为分析驱动人格参数持续优化
- 验证结果：100% MBTI 对齐准确率，支持 GPT-4/Llama/Qwen 全模型

**对本项目的启示**：

1. **人格演化不一定要用数值化规则**：evolving_personality 证明了"让 LLM 自己分析自己"是可行的。方案 A（记忆驱动特征演化）的"每 N 轮汇总 → LLM 分析 → 生成特征增量"流程，本质上就是反思机制
2. **连续权重 > 离散标签**：本项目如果采用 MBTI 或 OCEAN 维度，应使用连续值（0-1），而非离散类型（如"外向型/内向型"）
3. **主导功能保护**：人格演化时必须保护核心特质不被动摇，这与本项目"role_anchor 不可演化"的设计原则一致

### 9.2 OCEAN（Big Five）大五人格模型

**来源**：心理学学术界最广泛验证的人格框架（Costa & McCrae, 1992），被数千篇论文引用

**核心发现**：

五个独立正交维度，每个维度产生截然不同的角色声音：

| 维度 | 高值表现 | 低值表现 | 与人格独立性的关系 |
|------|----------|----------|----------|
| **O**penness（开放度） | 好奇、富有想象力、愿意尝试新观点 | 传统、务实、固执己见 | 低开放度=角色更坚持自己的世界观 |
| **C**onscientiousness（尽责性） | 自律、有条理、遵守规则 | 随性、不拘小节、灵活 | 高尽责性=角色对自己的原则更坚持 |
| **E**xtraversion（外向性） | 热情、社交、主动表达 | 内敛、安静、被动回应 | 高外向性=角色更主动发起话题和表达观点 |
| **A**greeableness（宜人性） | 善解人意、配合、回避冲突 | 直言不讳、坚持己见、不惧冲突 | **低宜人性=人格独立性的核心** |
| **N**euroticism（情绪稳定性） | 易焦虑、情绪波动大、敏感 | 情绪稳定、冷静、不易受影响 | 高神经质=角色容易被用户言行影响 |

**Agreeableness 与"人格自主性"的关键关系**：

这正是"人格独立性"的量化基础。**Agreeableness 越低，角色越独立、越不盲从**：

- **高 Agreeableness（0.7-1.0）**：使用软化语言（"也许你说得对..."），回避冲突，道歉频繁，倾向于配合用户
- **中 Agreeableness（0.3-0.7）**：有自己立场但愿意协商，选择性表达不同意见
- **低 Agreeableness（0.0-0.3）**：使用尖锐语言（"我不同意"），直接挑战，不在乎伤害感受，**这正是"独立人格"的极致表现**

**对本项目的启示**：

1. **用 OCEAN 替代 stance_matrix**：OCEAN 有学术验证、有明确的行为映射、有成熟的心理学量表，比自定义的 stance_matrix 更可靠
2. **角色卡设计可以借用 OCEAN 量表**：用户创建角色时，可以用简化的 OCEAN 问卷替代手动填写 stance_matrix，降低使用门槛
3. **人格演化有框架支撑**：OCEAN 的长期稳定性已被研究证实，成年后 OCEAN 变化缓慢但有规律可循，这为"人格成长性"的演化速度提供了参考（不应在短期内剧烈变化）
4. **Agreeableness 是"人格自主性"的直接度量**：不需要额外设计"异议触发条件"，低 Agreeableness 的角色自然会表达不同意见

### 9.3 FIVE Character Engine（开源 MCP Server）

**来源**：开源 MCP Server 项目，专注于用结构化约束控制 LLM 角色一致性

**核心发现**：

- 用 **JSON 人格约束**而非 System Prompt 来控制角色一致性
- 核心理念：**角色崩坏不是因为忘记了规则，而是输入绕过了规则**
- 解决方案：用**认知原语（cognitive primitives）**在输入层过滤，而非输出层审核
  - 传统方式：System Prompt 说"你应该这样表现" → LLM 可能在长对话中遗忘
  - FIVE 方式：在构建 prompt 时，根据人格约束**过滤和改写**用户输入，使 LLM 始终在角色认知框架内回应

**对本项目的启示**：

1. **输入过滤 > 输出审核**：可以借鉴"输入过滤"思路——在构建 System Prompt 时，不仅告诉 AI"你是谁"，还告诉 AI"面对什么类型的输入时应该怎么反应"
2. **结构化约束更可靠**：纯文本 System Prompt 可能被 LLM 忽略，JSON 格式的人格约束可以作为 prompt 构建时的"硬约束"（如：如果用户输入触及角色的 taboo_topics，则在 prompt 中明确标注"这个话题让你不舒服"）
3. **角色卡 = 输入处理协议**：角色卡不仅是"角色描述"，还可以定义"如何处理不同类型的用户输入"

### 9.4 SillyTavern 角色卡系统

**来源**：开源 AI 角色扮演前端项目（SillyTavern），拥有庞大的角色卡社区

**核心发现**：

- 支持"**矛盾者**"（contradictor）角色——一个专门负责质疑和反对的角色
- **角色卡 = 交互协议（interaction protocol）**，不只是描述性文本
- 用角色卡模拟"持不同意见的人"是**切实可行**的，社区中有大量成功案例
- 角色卡中包含"对话风格"、"反对触发条件"、"争论策略"等字段

**对本项目的启示**：

1. **"AI表达不同意见"已被实践验证**：SillyTavern 证明了让 AI 角色表达不同意见是可行的，只需要在角色卡中明确定义
2. **角色卡应包含"交互协议"**：本项目角色卡不应只是"角色描述"，还应包含"角色如何与用户互动"的协议层（如：什么时候该反对、用什么方式反对）
3. **"矛盾者"角色卡可以作为测试基准**：在开发人格自主性功能时，可以创建一个"矛盾者"测试角色卡，专门验证 AI 是否能坚持不同意见

### 9.5 CharacterGPT（NAACL 2025 论文）

**来源**：NAACL 2025 学术论文，研究 LLM 角色人格的动态演化

**核心发现**：

- 通过**章节摘要**逐步更新角色人格，而非逐轮更新
- 模仿人类**记忆巩固（schema consolidation）**过程：
  ```
  对话经历 → 短期记忆 → 定期汇总（章节摘要） → 长期记忆整合 → 人格快照更新
  ```
- 每次更新产生的是"人格快照"（persona snapshot），而非增量修改
- 关键创新：人格演化是**阶段性**的，不是连续的——人类也是在经历一段时期后才"发现自己变了"

**对本项目的启示**：

1. **"记忆巩固周期"比"固定轮数"更自然**：不是每 50 轮固定演化，而是根据对话内容的"章节边界"（如一个话题结束、一个关系阶段结束）来触发演化
2. **人格快照 > 增量修改**：每次演化生成的是"当前人格的完整描述"，而非"上次人格 + 0.1 温暖度"。这更符合人类"我变成了什么样的人"的自我认知方式
3. **与方案 A 高度一致**：方案 A 的"演化快照"概念与 CharacterGPT 的"章节摘要"思路一致，验证了方案 A 的学术合理性
