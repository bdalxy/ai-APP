## 2. 学习

> 用"是什么→为什么→怎么选"结构，供用户学习理解

### 2.1 什么是交叉审查？

交叉审查就是让不同专业方向的审查员（架构师、安全工程师、Python专家）同时审查同一个项目的不同方面，然后互相验证发现的问题。这样可以减少"盲点"——架构师可能漏掉安全问题，安全工程师可能漏掉架构问题，交叉审查让两者互补。

### 2.2 为什么需要全量审查？

项目经过多次迭代修改后，容易积累三类问题：
1. **残留问题**：旧代码没删干净（如旧版本号、旧颜色引用）
2. **不一致问题**：不同文件之间的约定不一致（如版本号在 build.gradle 和 strings.xml 中不同）
3. **连锁问题**：修一个问题时引入的新问题（如修改了 Python 函数签名但 Kotlin 端没同步）

全量审查就是逐文件排查这三类问题。

### 2.3 审查维度说明

| 维度 | 是什么 | 为什么重要 |
|------|--------|-----------|
| 编译正确性 | 代码能否通过编译 | 编译不过 = 功能完全不可用 |
| 线程安全 | 多线程访问共享数据是否安全 | 线程不安全 = 随机崩溃，难以复现 |
| 内存安全 | 是否有内存泄漏 | 内存泄漏 = APP越来越卡，最终崩溃 |
| 空安全 | null值是否被正确处理 | 空指针 = 闪退 |
| UI规范 | 颜色/字符串是否用资源引用 | 硬编码 = 无法换主题/国际化 |
| 接口一致性 | Kotlin调用Python的参数是否匹配 | 不一致 = 功能异常甚至崩溃 |
| 安全 | API Key、数据存储、日志 | 安全问题 = 用户数据泄露 |
| 异常处理 | 出错时是否有兜底 | 没兜底 = 一个错误导致整个功能挂掉 |
