#### 2.2 ⚠️ syncCharacterToPython() 与 loadCharacterCard() 不一致

**发现**：`MainActivity.kt` 中有两个方法构建角色卡 JSON 传给 Python：

| 方法 | 位置 | 发送字段数 | 发送的字段 |
|------|------|-----------|-----------|
| `loadCharacterCard()` | L848-865 | 11 个 | name, personality, speaking_style, backstory, greeting, emotional_tendency, self_identity, core_traits, taboo_topics, role_anchor, world_book_id |
| `syncCharacterToPython()` | L979-989 | 8 个 | name, personality, speaking_style, backstory, greeting, emotional_tendency, self_identity, world_book_id |

**缺失字段**：`syncCharacterToPython()` 未发送 `core_traits`、`taboo_topics`、`role_anchor`。

**影响**：角色切换后（CharacterSelectActivity 返回），Python 端不会更新这三个字段，AI 将使用旧角色的核心特质、禁忌话题和角色锚点，导致角色切换不完整。

**影响范围**：中等——用户切换角色后，核心特质/禁忌话题/角色锚点可能残留旧角色设置。

**修复建议**：在 `syncCharacterToPython()` 中补充 `core_traits`、`taboo_topics`、`role_anchor` 三个字段。

#### 2.3 WorldBookEntry — ✅ 已统一

**Kotlin**：`com.aicompanion.app.WorldBookEntry`（7 字段）
```kotlin
data class WorldBookEntry(
    val id: String, val category: String, val content: String,

```

**module 层**：`module/worldbook/WorldBookModule.kt:12`
```kotlin
typealias WorldBookEntry = com.aicompanion.app.WorldBookEntry
```

**Python**：`src/world_book/world_book.py:52-67`（14 字段）
```python
@dataclass

    id, keys, key_secondary, content, comment, constant, selective,
    probability, priority, regex_keys, case_sensitive, max_injections,
```

**分析**：Kotlin 端是 Python 端的简化子集，`WorldBookModuleImpl` 在 `jsonToEntry()`/`entryToJson()` 中做字段映射。Kotlin 端缺失的字段（`selective`, `probability`, `regex_keys`, `case_sensitive`, `max_injections`, `cooldown_rounds`, `constant`）在创建/编辑时会使用默认值。

**结论**：✅ 双重定义问题已通过 typealias 解决。Kotlin 端简化是设计选择（UI 层不需要全部字段），但限制了通过模块接口创建世界书条目时的灵活性。**低优先级**。

### 四、残留清理检查

#### 4.1 🟠 @Deprecated 代码在核心路径使用

**问题**：v3.0 修复时将 Kotlin 端插件体系（`PluginInfo`, `PluginType`, `PluginRegistry`, `BuiltinPlugins`, `IPlugin`）全部标记为 `@Deprecated`，但**这些代码仍在核心路径中被使用**：

| 文件 | 使用的废弃 API | 用途 |
|------|---------------|------|
| `AICompanionApp.kt:141-142` | `PluginRegistry.init()`, `BuiltinPlugins.registerAll()` | **应用启动时初始化** |
| `SettingsActivity.kt:152-159` | `PluginRegistry.getPluginCount()`, `getEnabledPluginCount()` | **设置页显示插件数量** |
| `PluginManageActivity.kt:79,176,178` | `PluginRegistry.getAllPlugins()`, `PluginInfo.toPluginItem()` | **插件管理页核心逻辑** |
| `PluginModuleImpl.kt:74,110-145,175-177` | `PluginRegistry.getAllPlugins()`, `PluginInfo.toMeta()` | **模块桥接层核心逻辑** |

**废弃的 4 个文件**：
- `plugin/IPlugin.kt` — 接口已废弃
- `plugin/BuiltinPlugins.kt` — 内置插件注册已废弃
- `plugin/PluginRegistry.kt` — 插件注册表已废弃
- `plugin/PluginInfo.kt` — 数据模型已废弃（由 `PluginItem` 替代）

**影响**：这些代码虽然标记为 `@Deprecated`，但**实际上仍被编译和运行**。废弃标记只是编译期警告，功能上仍在使用。如果未来移除这些文件，上述 4 个核心类将编译失败。

**分析**：这是因为 Kotlin 插件体系尚未完全迁移到 Python 插件体系。`PluginManageActivity` 通过 `PluginInfo.toPluginItem()` 桥接新旧模型，`PluginModuleImpl` 通过 `PluginInfo.toMeta()` 桥接。这些桥接代码是**过渡期设计**，但废弃标记暗示它们应该被移除。

**建议**：
- 明确废弃清理的时间线（如 v2.1）
- 将 `PluginManageActivity` 改为直接使用 Python 插件 API（`list_plugins()`, `toggle_plugin()`）
- 将 `SettingsActivity` 的插件计数改为调用 Python 的 `get_plugin_count()`
- 将 `AICompanionApp` 的插件初始化移至 Python 端（`init()` 中已调用 `_get_plugin_manager().load_all()`）
