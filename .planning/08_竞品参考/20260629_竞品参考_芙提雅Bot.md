## 一、知识库 vs 世界书

### 对比

| 维度 | 芙提雅 Bot | AI Companion |
|------|-----------|-------------|
| **名称** | 尘白知识库 | 世界书（World Book） |
| **技术** | RAG 向量检索（AstrBot 内置） | 关键词匹配 + 条目注入 |
| **内容** | 尘白禁区游戏设定、角色资料 | 两个独立世界书：三次元现实（10条，常驻）+ 二次元认知（8条，可选开关） |
| **检索方式** | 混合检索（Dense + BM25） | 触发词匹配（keys 字段） |
| **文档支持** | PDF/DOCX/Markdown 上传 | 纯 JSON 条目 |
| **注入时机** | 对话时自动检索 | 触发词命中时注入 System Prompt |
| **管理** | WebUI 可视化管理 | Android 原生 UI（WorldBookActivity） |

### 分析

**本质上是同一类功能**——都是"给 AI 提供背景知识"。但实现路径不同：

- **他们**走的是标准 RAG 路线：文档上传 → 切片 → 向量化 → 检索 → 注入。优点是可处理任意文档，缺点是检索质量依赖 embedding 模型，回答可能不够精准。
- **我们**走的是触发词匹配路线：预设条目 + 关键词触发 → 精确注入。优点是精准可控、体积小，缺点是需要人工编写条目，无法自动消化大量文档。

**结论**：当前我们的世界书已经足够（`方案_默认角色卡与世界书.md` 中已设计两个世界书：三次元现实 10 条 + 二次元认知 8 条，共 18 条，覆盖日常对话场景）。短期不需要引入 RAG 向量检索，但 v2.1 可考虑加入"文档上传 → 自动生成世界书条目"的功能，降低用户编写门槛。

**行动建议**：v2.0 保持现状，v2.1 做文档导入。

## 二、小游戏

### 芙提雅 Bot 的"小游戏"

视频简介中提到"各种小游戏"，可能指:
- AstrBot 插件生态中的游戏类插件（猜数字、成语接龙、文字冒险等）
- 利用 IM 的文本交互特性，做基于聊天的轻量游戏

### 我们的现状

项目中**完全没有小游戏模块**，需求文档中也未涉及。

### 分析

小游戏是一种**增强用户粘性和互动趣味性**的手段。对于 AI 陪伴类应用，适度的游戏化可以：
- 在对话冷场时提供互动入口
- 增加每日打开 APP 的理由
- 让 AI 角色"更有趣"

但需要注意：
- 我们不是游戏平台，游戏是附加功能，不是核心
- 开发成本需要评估——即使是简单的文字游戏，也需要额外的对话逻辑处理

### 建议

| 优先级 | 内容 | 说明 |
|:---:|------|------|
| P2 | 内置文字小游戏 | 猜数字、成语接龙、真心话大冒险等文本交互游戏 |
| P3 | 世界书驱动的"角色扮演场景" | 通过世界书条目定义场景规则，AI 扮演游戏主持人 |

**暂不纳入 v2.0，等核心功能稳定后再考虑。**

根据 `品质打磨实施方案.md`，当前 UI 存在以下已知问题：

| 编号 | 问题 | 严重度 |
|:---:|------|:---:|

| UX-05 | 缺少触觉反馈 | P2 |

| UX-07 | 消息发送无乐观更新 | P1 |
| UX-08 | 无障碍支持不完整 | P2 |

### 芙提雅 Bot 的 UI 特点

芙提雅 Bot 寄生在 QQ 中，UI 就是 QQ 聊天界面。优点是用户零学习成本，缺点是：
- 完全受限于 QQ 的 UI 框架
- 无法做沉浸式视觉设计
- 无法展示角色形象

### 我们的差异化空间

这正是我们**独立 APP 的最大优势**——可以做对方做不到的事：

1. **角色形象展示**：主界面可以有角色立绘、Live2D 动画
2. **沉浸式视觉**：深色玻璃态 + 樱羽主题，可以有独特的视觉调性
3. **自定义交互**：手势、动画、过渡效果，不受 IM 框架限制
4. **语音通话界面**：可以设计专门的语音通话 UI，而不是 QQ 语音消息

### 建议

v2.0 品质打磨阶段应优先修复上述 UX 问题（已列入计划），但 v2.1 应该考虑：

| 优先级 | 改进项 | 说明 |
|:---:|------|------|
| P1 | 修复已知 UX 问题（8项） | v2.0 已规划 |
| P1 | 角色形象展示区 | 主界面顶部增加角色头像/立绘展示区域 |
| P2 | 消息气泡动画 | 发送/接收消息的微动画 |
| P2 | 主题升级 | 从"看得下去"提升到"觉得好看" |
| P3 | Live2D 角色动画 | 长期目标，技术复杂度高 |

## 四、语音对话对比

### 双方技术栈

| 维度 | 芙提雅 Bot | AI Companion（当前） | AI Companion（VITS替换后） |
|------|-----------|---------------------|--------------------------|
| **TTS 引擎** | 未知（可能调 API 或系统 TTS） | Sherpa-ONNX Matcha | Sherpa-ONNX VITS |
| **TTS 音质** | 估计中等（实时语音） | 机械感较强，缺乏自然语调 | 预期提升 1-2 档 |
| **STT（语音识别）** | 有（实时语音对话） | 有（VoiceRecorder + MediaRecorder） | 同 |
| **实时对话** | 支持（实时语音对话） | 不支持（录音→发送→播放） | 不支持 |
| **离线** | 依赖服务器 | 完全离线 | 完全离线 |
| **多音色** | 不确定 | 单一音色 | 174 人可选（AISHELL3） |
| **中英混读** | 不确定 | 支持 | 方案 A 弱，方案 B 强 |

### 关键差距：实时语音对话

芙提雅的"实时语音对话"意味着：
- 用户说话 → AI 实时听取 → AI 实时语音回复
- 类似打电话，不是"录音→发送→等待→播放"
- 需要 ASR + LLM + TTS 三者低延迟串联

我们目前是**异步模式**：录音 → 发送语音消息 → AI 文字回复 → TTS 朗读。这更接近"语音消息"而非"语音通话"。

### 我们的优势

- **纯离线 TTS**：不需要网络，不需要 API 费用
- **多音色**：VITS-AISHELL3 有 174 个说话人，可以给不同角色配不同声音
- **跨平台模型**：.ttsmodel.zip 格式，PC 训练 → 移动端使用

### 建议

| 阶段 | 内容 | 说明 |
|:---:|------|------|
| v2.0 | VITS 替换 Matcha（方案 A） | 零代码改动，音质提升，174 音色 |
| v2.1 | 多模型切换 + 模型导入导出 | 用户可选不同音色 |
| v2.2 | 实时语音对话 | 需要 ASR 流式 + TTS 流式 + 低延迟调度 |

**实时语音对话是 v2.2 的远期目标**，当前先做好 TTS 音质提升。

## 五、独立 APP 基础上对接 QQ 机器人

### 与底层原则的冲突分析

> **重要**：对接 QQ/微信 Bot 与项目底层原则存在冲突，需要审慎评估。

| 原则 | 冲突点 | 严重度 |
|------|------|:---:|
| 纯本地优先 | Bot 需要稳定在线的服务端，手机 APP 无法满足 | 🔴 |
| 纯本地优先 | QQ 协议需要外部服务器中转（OneBot/反向 WS） | 🟡 |
| 平台范围 | 当前仅 Android + PC，Bot 对接属于新平台扩展 | 🟡 |
| 免费开源 | 用户可能需要付费的 QQ 机器人框架或服务器 | 🟡 |

**核心矛盾**：Bot 的本质是一个"服务端"——它需要 24 小时在线、有固定 IP 或 WebSocket 连接。而我们的核心原则是"纯本地，不建设后端服务器"。手机 APP 天然不适合做 Bot 服务端（后台限制、耗电、网络不稳定）。

### 技术可行性

1. **方案 A：APP 内自建 Bot 桥接**
   - 在 APP 内运行 HTTP/WS 服务 → 对接 OneBot 协议 → 连接 QQ
   - **问题**：Android 后台限制严格，APP 被杀死后 Bot 掉线
   - **问题**：QQ 的反机器人检测越来越严格，非官方客户端有封号风险
   - **与原则冲突**：需要持续的互联网连接和后台服务

2. **方案 B：嵌入 AstrBot**
   - 通过 Chaquopy 在 Android 上运行 AstrBot
   - **问题**：AstrBot 依赖 Docker、WebUI、数据库等，几乎无法嵌入
   - **结论**：不可行

3. **方案 C：PC 端 Bot**
   - PC 运行 Bot 服务（AstrBot 或自建），手机 APP 通过局域网连接
   - **优势**：PC 可以 24 小时运行，网络稳定
   - **优势**：共享同一套角色卡/世界书/记忆
   - **与原则兼容**：Bot 运行在用户自己的 PC 上，不涉及第三方服务器
   - **问题**：需要 PC 开机，PC 版还没做

### 对比分析

| 维度 | 方案 A（APP内桥接） | 方案 B（嵌入 AstrBot） | 方案 C（PC端 Bot） |
|------|:---:|:---:|:---:|
| 开发量 | 中等 | 高 | 低（复用开源） |
| 维护成本 | 中等 | 高 | 低 |
| 稳定性 | 差（后台被杀） | 差（无法嵌入） | 好（PC 24h运行） |
| 封号风险 | 有 | 有 | 有 |
| 与原则兼容 | 部分冲突 | 冲突 | 基本兼容 |
| 实现周期 | 2-3 周 | 不可行 | 1-2 周 |

### 额外风险：QQ 封号

所有方案都面临同一个风险：**QQ 对非官方 Bot 的打击越来越严**。即使用 OneBot 协议、即使用签名服务，QQ 账号仍有被封禁的风险。芙提雅 Bot 作为商业项目可能有官方合作或白名单，这是我们不具备的。

### 建议

| 阶段 | 内容 |
|:---:|------|
| 远期 | 方案 C：等 PC 版开发后，作为 PC 版的附加功能评估 |
| 当前 | **不纳入任何近期计划**，优先级低于角色卡/世界书/插件/TTS 等核心模块 |

**理由**：
- Bot 对接与"纯本地优先"原则存在根本性矛盾
- QQ 封号风险是用户无法接受的代价
- 独立 APP 本身就是最好的交互载体——不需要寄生在 QQ 里
- 如果用户想在 QQ 里用 AI 聊天，芙提雅 Bot 已经提供了这个选项

## 六、记忆系统对比

### 双方能力

| 维度 | 芙提雅 Bot（AstrBot） | AI Companion |
|------|----------------------|-------------|
| **长期记忆** | 有（社区插件） | 有（VectorStore + 混合检索） |
| **好感度系统** | 有（affinity system） | 无 |
| **用户画像** | 有（user profiling） | 有（Analyzer 用户画像） |
| **自学习** | 有（self-learning） | 无 |
| **记忆提取** | 依赖 LLM | 规则模式 + LLM 模式 + 节流 |
| **记忆检索** | 向量检索 | 倒排预过滤→余弦精排→BM25→时间衰减 |
| **记忆衰减** | 不确定 | 指数衰减 + 频率加权 + 半衰期可配 |
| **记忆合并** | 不确定 | 相似检测 + 冲突解决 + 自动合并 |
| **记忆归档** | 不确定 | LLM 摘要 + 规则摘要 |
| **记忆可视化** | WebUI 管理面板 | Android 原生 UI（列表/详情/编辑/删除） |
| **备份恢复** | 不确定 | 完整备份 + JSON + 增量 + 校验 |
| **关系图** | 无 | 记忆关系图 + 标签系统 |
| **变更日志** | 不确定 | 完整审计追踪 |

### 分析

我们当前的记忆系统（P4）**功能相当完整**，从需求文档来看有 20+ 个子功能。对比 AstrBot 的社区插件，我们的记忆系统在以下方面更强：

- **记忆衰减**：指数衰减 + 频率加权，AstrBot 插件可能没有
- **记忆合并**：自动去重和冲突解决
- **记忆分析**：趋势分析、主题聚类、用户画像
- **审计追踪**：完整的变更日志

但在以下方面有差距：

- **好感度系统**：AstrBot 有专门的 affinity system，我们没有
- **自学习**：AstrBot 声称支持 self-learning，我们没有

### 建议

| 优先级 | 内容 | 说明 |
|:---:|------|------|
| P2 | 好感度系统 | 基于对话频率、情感倾向、互动深度计算好感度，影响 AI 的语气和主动程度 |
| P3 | 自学习 | 从用户反馈中自动调整回复风格，但技术复杂度高，需谨慎设计 |

**好感度系统值得在 v2.1 加入**——它和主动推送（P5）配合，可以让 AI 更"像人"。自学习暂缓。

## 总结：我们的定位与差异化

```
                    芙提雅 Bot                    AI Companion
                    ──────────                    ────────────
产品形态        IM 寄生（QQ）              独立 APP
技术栈          Python (AstrBot)           Kotlin + Chaquopy Python
角色            版权角色（有风险）          完全原创（星遥）
隐私            数据在服务器               数据完全本地
UI              受限于 QQ 界面             自定义沉浸式 UI
语音            实时对话（网络依赖）        离线 TTS（多音色）
知识库          RAG 文档检索               触发词世界书
记忆            社区插件                   自研完整系统（20+功能）
小游戏          有                         无
插件生态        1000+                      自研插件系统（刚起步）
多平台          QQ/微信/Telegram 等        Android + PC（规划中）
开源            AGPL-v3                   计划开源
```

### 我们的核心差异化

1. **隐私第一**：完全本地，数据不出设备
2. **沉浸式体验**：独立 APP 的视觉和交互自由度远超 IM 寄生
3. **版权安全**：原创角色，无法律风险
4. **记忆系统深度**：衰减、合并、分析、审计——比社区插件更完整
5. **离线 TTS**：不依赖网络，不消耗 API 额度

### 需要追赶的

1. **实时语音对话**（远期目标，技术门槛高）
2. **好感度系统**（v2.1 建议加入）
3. **UI 品质提升**（v2.0 修复已知问题，v2.1 视觉升级）
4. **Agent 能力**（MCP/工具调用，远期参考 AstrBot 设计）
