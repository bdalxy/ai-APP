## 三、v2.1.0 "生态扩展" 预规划

### 3.1 核心目标

将 AI Companion 从"单人使用"升级为"生态兼容"——支持导入导出社区角色卡和世界书，完善语音情感表达。

### 3.2 主要方向

| 方向 | 内容 | 关联 |
|------|------|------|
| **角色卡生态** | Character Card V2 导入导出、tavo 格式兼容、内容迁移工具（世界书→角色卡） | 六方向 R5 |
| **世界书生态** | 自定义世界书创建 UI、条目冲突标记、导入导出 | 六方向 W4 |
| **插件生态** | .aiplugin.zip 导入导出、6 个钩子节点完整实现 | 六方向 P5 |
| **语音升级** | GPT-SoVITS 情感语音合成（用户已在 PC 上使用）、音色混合、逐条朗读 | 六方向 V3 |
| **导航评估** | 功能增多后评估是否需要底部导航栏 | 六方向 U1 |

### 3.3 暂缓项目

| 项目 | 原因 |
|------|------|
| 插件市场 | 用户基数不足，暂不建设 |
| 真正的"同时对话"（Phase 2-3） | 复杂度高，Phase 1 消息队列先行 |
| 长对话摘要压缩 | 类微信/QQ 短对话为主，长对话少 |
| 端侧本地大模型 | 存储占用大、设备要求高、量化模型智能不足 |

## 四、v2.2.0 "PC扩展" 远景

### 4.1 核心目标

发布 Windows 桌面版，实现 Android ↔ PC 双端数据同步。

### 4.2 技术方案（待评估）

| 选项 | 方案 | 优势 | 劣势 |
|------|------|------|------|
| A | Kotlin Multiplatform (Compose Desktop) | 代码复用率高 | 生态不如 Electron |
| B | Electron + Python 后端 | 生态丰富、UI 灵活 | 体积大、性能一般 |
| C | Python + Qt (PySide6) | 原生性能、Python 代码复用 | UI 开发效率低 |

**推荐**：待 v2.1 完成后评估。

### 4.3 同步方案

- 文件导入导出（JSON 格式）
- 本地网络同步（同一 WiFi 下）
- 不建设云端同步服务（纯本地优先原则）

## 七、讨论要点

1. **v2.0 的"完成"定义**：当前功能层面 95% 完成，但质量层面（硬编码、锁优化、ChatViewModel 拆分）仍需收尾。是现在发布 v2.0.0 正式版还是等质量修复完成？

2. **v2.1 的启动时机**：v2.0 P0/P1 修复完毕后立即启动 v2.1，还是给 v2.0 留一段稳定期？

3. **GPT-SoVITS 的集成方式**：用户已在 PC 上使用 GPT-SoVITS。是否先在 PC 端测试集成，再移植到 Android？

4. **角色卡导入的优先级**：用户计划从 tavo 导入角色卡。是否优先做 tavo 兼容？

5. **PC 版技术选型**：Kotlin Multiplatform vs Electron vs Python Qt，何时启动评估？
