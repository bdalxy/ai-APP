## 二、各方向现状与关键问题

### 2.1 聊天系统（主干）

**现状**：
- ChatViewModel + ConversationSessionManager 实现多会话流式对话
- Python chat_engine/ + chat_bridge/ 负责 System Prompt 构建和 API 调用
- 当前 `build_system_prompt()` 仅做角色卡文本平铺，没有"特质 × 情境 × 价值观"的三因素注入
- 支持搜索、导出、流式取消

**关键问题**：

| # | 问题 | 影响范围 | 用户反馈与调研结论 |
|:---:|------|------|------|
| C1 | **人格自主性如何注入** | 角色卡 → 聊天系统 | 角色卡 OCEAN 五维度 + 话题激活矩阵 → System Prompt 行为指令。角色卡赋予 AI 人格，AI 据此像真人一样决定"是否反对"——反对与否由角色卡决定，不由破限决定 |
| C2 | **上下文窗口管理** | 聊天系统自身 | 本项目是类微信/QQ 式短对话，长对话相对少。采用"**滑动窗口为主 + 关键轮次保留**"混合策略。Token 预算：角色卡 30% + 世界书 15% + 历史消息 55%。短对话默认保留最近 20 轮，长对话自动压缩为摘要 |
| C3 | **世界书注入时机** | 世界书 → 聊天系统 | 世界书本质是**知识库**。分类处理：**世界观类**始终注入（默认已知），**事物/技能类**触发匹配注入，**事件类**仅在相关场景注入。触发匹配结果缓存当前轮次，不重复计算 |
| C4 | **插件钩子** | 插件系统 → 聊天系统 | 不同插件和世界书介入位置不同。参考 SillyTavern 和 tavo，定义 6 个标准钩子节点（见 2.4 节） |

#### 2.1.1 上下文窗口管理方案（樱羽特色）

**设计原则**：本项目是类微信/QQ 式短对话，不需要复杂的滑动窗口 + 摘要压缩双引擎。采用**轻量级混合方案**。

```
┌──────────────────────────────────────────────────────────────┐
│  Token 预算分配（以 deepseek-v4-flash 128K 为例）              │
│                                                              │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐│
│  │ 角色卡 + 人格  │  │ 世界书知识     │  │ 对话历史（滑动窗口）    ││
│  │ 自主性指令     │  │ 常量 + 触发     │  │                      ││
│  │ ~30% (38K)   │  │ ~15% (19K)    │  │ ~55% (71K)           ││
│  └──────────────┘  └──────────────┘  └──────────────────────┘│
│                                                              │
│  ┌─ 短对话模式（< 50轮）─────────────────────────────────┐   │
│  │ 保留全部历史消息，不做压缩                               │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  ┌─ 长对话模式（≥ 50轮）─────────────────────────────────┐   │
│  │ 保留最近 20 轮完整消息 + 前 30 轮 LLM 摘要（每 10 轮一摘要）│   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

**特色设计**：由于是类微信/QQ 短对话，"关键轮次保留"比"全量摘要"更重要。将以下轮次标记为"不可压缩"：
- 角色第一次表达不同意见的轮次
- 情绪强度 > 0.7 的轮次
- 用户明确标记为"重要"的轮次（未来功能）

#### 2.1.2 类微信/QQ 即时通讯风格

**用户需求**：AI 和用户能像微信/QQ 那样**同时互相发送消息**（非阻塞式对话）。

**技术分析**：

| 方面 | 传统方案（阻塞式） | 即时通讯风格（非阻塞式） |
|------|:---:|:---:|
| 用户发送消息 | 等待 AI 回复完成才能再发 | 可连续发送多条，AI 逐条回复 |
| AI 回复中 | 输入框禁用 | 输入框始终可用 |
| 消息队列 | 单条流 | 用户消息队列 + AI 回复队列并行 |
| 实现复杂度 | 低 | **中高** |

**推荐方案**：**分阶段实现**

| 阶段 | 内容 | 复杂度 |
|:---:|------|:---:|
| Phase 1（MVP） | 用户可在 AI 回复时继续输入和发送，消息进入队列，AI 逐条回复 | 中 |
| Phase 2（增强） | 用户发送多条后，AI 智能判断哪些可以合并回复，哪些需要分别回复 | 中高 |
| Phase 3（完整） | 真正的"同时对话"——AI 在回复过程中，用户插入新消息，AI 感知并调整回复方向 | 高 |

**Phase 1 实现方案**：
```
用户连续发送 [msg1, msg2, msg3]
    ↓
消息队列: [msg1, msg2, msg3]
    ↓
AI 逐条处理:
  msg1 → API 调用 → 回复 A1
  msg2 → API 调用（含 A1 上下文）→ 回复 A2
  msg3 → API 调用（含 A1, A2 上下文）→ 回复 A3
```

### 2.2 角色卡（人格层）

**现状**：
- CharacterData 包含 personality、core_traits、taboo_topics、emotional_tendency、role_anchor、self_identity、ocean_profile
- CharacterModule 提供加载/保存/同步角色卡
- **默认仅 1 个角色"星遥"**（修正：之前文档中"3 个原创角色"的说法有误，除默认角色外，月影/晨风/霜羽为可选预设角色，非默认角色卡）

**关键问题**：

| # | 问题 | 影响范围 | 用户反馈与调研结论 |
|:---:|------|------|------|
| R1 | **人格自主性的角色卡落地** | 角色卡自身 | 人格自主性 = 特质 × 情境 × 价值观（已在人格设计文档中完成）。角色卡新增 `ocean_profile`（5维）+ `topic_activation`（话题激活矩阵）+ `value_boundaries`（价值观底线） |
| R2 | **交互协议层** | 角色卡 → 聊天系统 | 参考 SillyTavern 的"矛盾者"角色卡设计。交互协议 = 角色如何与用户互动的规则。**建议放在角色卡内**，作为角色卡的一个字段（如 `interaction_protocol`），而非独立于角色卡 |
| R3 | **角色卡与破限解耦** | 角色卡 → 插件系统 | **破限默认启用、内置入项目底层**（非插件化）。破限保障"对话自由"（全局），角色卡驱动"人格自主性"（角色特定）。两者正交：破限让 AI 能说什么，角色卡决定 AI 想说什么 |
| R4 | **人格成长性** | 角色卡 → 聊天系统 | 反对/不反对**由角色卡决定**——AI 依据角色卡获得人格，像真正的人一样去决定。已在人格设计文档中完成 A+B 融合方案（OCEAN 数值框架 + LLM 反思机制） |
| R5 | **角色卡导入导出** | 角色卡自身 | 支持**大众化格式**（Character Card V2 PNG 嵌入 JSON）。用户计划从 tavo 导入角色卡和世界书。需要设计字段映射 + 内容迁移工具（世界书中的"人设"内容 → 角色卡字段） |

#### 2.2.1 角色卡格式兼容方案

**支持的导入格式**：

| 格式 | 来源 | 说明 |
|------|------|------|
| Character Card V2 | Chub.ai / JanitorAI 社区 | PNG 嵌入 JSON，最通用的社区格式 |
| SillyTavern JSON | 酒馆 | 角色卡 JSON 文件 |
| tavo 角色卡 | tavo | 用户计划导入，需要字段映射 |
| 本项目原生格式 | 樱羽 | 内部格式，含 OCEAN + 话题激活矩阵 + 价值观底线 |

**字段映射表**（tavo → 樱羽）：

| tavo 字段 | 樱羽字段 | 映射方式 |
|------|------|------|
| name | name | 直接映射 |
| description / personality | personality | 合并映射 |
| first_mes | greeting | 直接映射 |
| scenario | backstory | 合并映射 |
| mes_example | speaking_style | 合并映射 |
| creator_notes | — | 忽略或存入 comment |
| （无） | ocean_profile | 默认 0.5（中性） |
| （无） | topic_activation | 默认空（使用系统默认） |
| （无） | value_boundaries | 默认空（无特殊底线） |

**内容迁移工具**：用户提到"主要人设写在世界书里，但实际上应该在角色卡里"。需要一个工具：
1. 扫描世界书条目，标记"疑似人设内容"（如"XX是一个温柔的人"）
2. 用户确认 → 移动到角色卡对应字段
3. 世界书中的原条目删除或标记为"已迁移"

### 2.3 世界书（知识库）

**现状**：
- WorldBookModule + 触发词匹配（BM25）
- 两套世界书：三次元现实（常驻，9条）+ 二次元认知（可选开关，8条）
- 角色卡通过 `worldBookId` 绑定世界书，切换角色时自动切换

**核心理念更新**：世界书本质是**知识库**——AI 已经理解的东西。不是"触发时才注入的工具"，而是"AI 拥有的知识体系"。

**关键问题**：

| # | 问题 | 影响范围 | 用户反馈与调研结论 |
|:---:|------|------|------|
| W1 | **知识检索精度** | 世界书自身 | BM25 对短对话场景足够。v2.0 不引入 RAG，v2.1 评估是否需要。当前重点是优化触发词匹配的**分类化**——不同类别用不同匹配策略 |
| W2 | **角色卡价值观 vs 世界书知识** | 角色卡 ↔ 世界书 | 角色卡赋予 AI 人格，三观/价值观受所处环境影响——**环境属于世界书范围**。冲突时：角色卡人格 > 世界书知识（人高于环境），但世界书中的"环境常识"（如"这个世界存在魔法"）是角色认知的基础 |
| W3 | **世界书注入格式** | 世界书 → 聊天系统 | 世界书条目以自然语言段落注入 System Prompt。注入位置：世界观类在角色卡描述前（"这个世界是..."），事物/技能类在角色卡描述后（"你知道..."），事件类在对话历史前（"当前环境中..."） |
| W4 | **自定义世界书** | 世界书自身 | 条目结构沿用现有 WorldBookEntry。需要"冲突标记"标注与角色卡价值观的冲突。用户不熟悉此领域，需要简化设计 |
| W5 | **世界书与插件联动** | 世界书 → 插件系统 | 时间天气等环境信息以"临时世界书条目"形式注入，统一放在知识库中管理。不同插件和世界书的介入位置不同——有些直接关联（如天气 → 世界观），有些相对独立（如翻译插件） |

#### 2.3.1 世界书分类体系（樱羽知识库）

**核心设计**：世界书 = 知识库，分为四大类别，每类有不同的注入策略。

```
┌─────────────────────────────────────────────────────────────────┐
│                    樱羽知识库 (Sakura Knowledge Base)              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐       │
│  │ 1.世界观知识    │  │ 2.事物知识     │  │ 3.技能知识     │       │
│  │ (Worldview)   │  │ (Entities)    │  │ (Skills)      │       │
│  │               │  │               │  │               │       │
│  │ 世界的基本规则   │  │ 人物/地点/物品  │  │ 角色能做什么    │       │
│  │ 常驻注入        │  │ 触发匹配注入    │  │ 触发匹配注入    │       │
│  │ "这个世界有魔法" │  │ "XX是YY的朋友" │  │ "XX会弹钢琴"  │       │
│  └───────────────┘  └───────────────┘  └───────────────┘       │
│                                                                 │
│  ┌───────────────┐  ┌───────────────┐                           │
│  │ 4.事件知识     │  │ 5.环境感知     │  ← 时间天气插件注入       │
│  │ (Events)      │  │ (Environment) │                           │
│  │               │  │               │                           │
│  │ 正在发生的事    │  │ 时间/天气/季节  │                           │
│  │ 场景触发注入    │  │ 自动注入        │                           │
│  │ "今天是XX节日" │  │ "现在是夏天"    │                           │
│  └───────────────┘  └───────────────┘                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**各类别的注入策略**：

| 类别 | 注入时机 | 优先级 | 示例 |
|------|:---:|:---:|------|
| **世界观知识** | 始终注入（System Prompt 固定段落） | 最高 | "这个世界存在魔法"、"时间流速与现实不同" |
| **事物知识** | 触发匹配注入（关键词/正则） | 中 | "星遥是月影的姐姐"、"星夜庭院位于城市边缘" |
| **技能知识** | 触发匹配注入（关键词/正则） | 中 | "星遥擅长弹钢琴"、"月影精通剑术" |
| **事件知识** | 场景触发注入（时间/地点/上下文） | 低 | "今天是中秋节"、"外面正在下雨" |
| **环境感知** | 自动注入（时间天气插件） | 动态 | "当前时间：下午3点，天气：晴，28°C" |

**与角色卡的关系**：
- 角色卡 = 人格（"我是谁"）
- 世界书 = 知识（"我知道什么"）
- 三观/价值观 = 角色卡人格 × 世界书环境（人格是内因，环境是外因，价值观是两者交互的结果）

### 2.4 插件系统（扩展层）

**现状**：
- PluginModule + ModuleRegistry + ModuleEventBus
- 破限插件（JailbreakPlugin）为唯一已实现插件
- 插件格式 .aiplugin.zip（v2.1 导入导出）

**已决策的重大变更**：

| 决策 | 结论 |
|------|------|
| **破限定位于 = 内置** | 破限不是插件，是**项目底层基础保障**。默认启用，始终生效。理由：本地存储 = 私密对话 = 不需要审查。目标是让 AI 像真人一样自由交流，不是为了违法乱纪 |
| **破限保障能说什么，人格定义想说什么** | 破限 = 全局"对话自由保障"（话题不设限）。角色卡 = 角色特定"人格自主性"（态度和立场有边界）。两者正交 |

**关键问题**：

| # | 问题 | 影响范围 | 用户反馈与调研结论 |
|:---:|------|------|------|
| P1 | **破限内置化** | 插件系统 → 聊天系统 | **已决策：内置入项目底层，非插件化**。从 PluginModule 中移出，作为聊天系统的内置层。在 System Prompt 构建时始终注入 |
| P2 | **破限与人格自主性解耦** | 插件系统 → 角色卡 | 破限 Level 4-5 的人格指令拆分到角色卡驱动。角色卡的人格自主性在所有场景都生效（因为破限始终启用） |
| P3 | **插件钩子体系** | 插件系统 → 聊天系统 | 参考 SillyTavern 和 tavo，定义 6 个标准钩子节点。不同插件在不同节点介入 |
| P4 | **成人内容策略** | 插件系统 → 角色卡 | 默认支持，不设开关。仅通过角色卡 taboo_topics 让用户自行限制。融入项目底层 |
| P5 | **插件生态** | 插件系统自身 | v2.1 导入导出。插件市场暂且搁置。第三方安全沙箱需要设计 |

#### 2.4.1 插件钩子体系（樱羽特色）

**6 个标准钩子节点**：

```
用户发送消息
    │
    ▼
[H1] pre-prompt-build  ← 插件可修改 System Prompt（如注入临时知识）
    │
    ▼
[H2] pre-api-call       ← 插件可修改 API 请求参数（如调整 temperature）
    │
    ▼
  API 调用 → AI 回复
    │
    ▼
[H3] post-response      ← 插件可修改 AI 回复内容（如翻译、过滤）
    │
    ▼
[H4] pre-ui-render      ← 插件可在 UI 渲染前处理（如敏感词替换）
    │
    ▼
  UI 渲染（用户看到回复）
    │
    ▼
[H5] on-message-sent    ← 消息发送完成后触发（如日志、统计）
    │
    ▼
[H6] on-session-change  ← 会话切换时触发（如切换世界书、重置状态）
```

**各个钩子的介入示例**：

| 钩子 | 谁介入 | 做什么 |
|------|------|------|
| H1 pre-prompt-build | 世界书、时间天气插件、记忆插件 | 注入知识、环境信息、检索记忆 |
| H2 pre-api-call | 防重复插件、参数调优插件 | 调整 penalty、添加句式约束 |
| H3 post-response | 翻译插件、内容过滤插件 | 翻译回复、过滤敏感内容 |
| H4 pre-ui-render | 表情包插件、Markdown 渲染插件 | 替换文本为表情包、格式化 |
| H5 on-message-sent | 日志插件、统计插件、日记插件 | 记录对话、触发角色自省日记 |
| H6 on-session-change | 世界书切换插件、角色卡切换插件 | 更新知识库、更新人格 |

**不同插件介入位置不同**：
- 有些插件直接关联进聊天流程（如世界书在 H1 介入）
- 有些插件相对独立（如翻译插件在 H3 介入，不影响其他流程）
- 插件通过 `hooks` 字段声明自己介入哪些节点

**实现方式**：基于现有 `ModuleEventBus` 扩展，定义 `DialogueEvent` 类型，插件订阅感兴趣的事件。**推荐方案 A（基于事件总线）**，耦合度低，可扩展性强。

### 2.5 语音系统（输出层）

**现状**：
- TtsModule + SherpaTtsEngine（支持 VITS-aishell3 + Matcha-baker 双模型）
- VITS-aishell3：174 人中文女声，模型文件约 116MB
- 本地 TTS，流式句子朗读
- 语音识别（ASR）已集成

**用户需求**：

| 需求 | 说明 |
|------|------|
| **音色混合** | 不同角色（星遥/月影/晨风/霜羽）有不同音色 |
| **声调调整** | 音调可调，不同情境用不同声调 |
| **情感表达** | 根据消息内容/情境，语音带有相应的情感色彩 |
| **逐条朗读** | 点击某条消息按钮，单独朗读该条消息 |

**关键问题**：

| # | 问题 | 影响范围 | 调研结论 |
|:---:|------|------|------|
| V1 | **音色与角色匹配** | 语音系统 → 角色卡 | 音色应作为角色卡字段（`voice_id`）。不同角色（星遥的温柔 vs 月影的清冷）对应不同 VITS 音色。角色卡中指定 `voice_id`，切换角色时自动切换音色 |
| V2 | **音色混合与声调调整** | 语音系统自身 | VITS 通过 Sherpa-ONNX 调用。Sherpa-ONNX 支持 `speed`（语速）参数。音色混合（voice blending）在 VITS 模型中不原生支持，但可以通过**选择不同 speaker_id** 实现"不同音色"。声调调整（pitch）在 Sherpa-ONNX 中有限支持，需要进一步测试 |
| V3 | **情感语音合成** | 语音系统自身 | **已决策：v2.1 目标模型为 GPT-SoVITS**（用户已在PC上使用过）。GPT-SoVITS 支持少样本音色克隆 + 情感控制，是理想的情感语音方案。当前 VITS 阶段通过语速调节模拟情感（开心=快，悲伤=慢），v2.1 升级到 GPT-SoVITS 实现真正的情感控制 |
| V4 | **逐条消息朗读** | 语音系统 → 聊天系统 | 在消息气泡旁添加**朗读按钮**（喇叭图标），点击后对该条消息单独调用 TTS 朗读。技术实现：在 ChatAdapter 中为每条消息添加点击事件，触发 SpeechManager.playSingleMessage() |
| V5 | **多音色管理** | 语音系统自身 | 174 种音色需要预览功能。在角色卡编辑页提供"音色试听"按钮，播放该音色的示例语音 |

#### 2.5.1 语音系统方案（樱羽特色）

**情感语音映射表**：

| 情感 | 语速 | 音量 | 声调倾向 | 适用场景 |
|------|:---:|:---:|:---:|------|
| 开心 | 1.2x | 正常 | 偏高 | 积极消息、问候 |
| 平静 | 1.0x | 正常 | 正常 | 日常对话 |
| 悲伤 | 0.8x | 偏低 | 偏低 | 负面消息、安慰 |
| 温柔 | 0.9x | 偏低 | 正常 | 亲密对话 |
| 严肃 | 1.0x | 正常 | 偏低 | 正式话题 |
| 兴奋 | 1.3x | 偏高 | 偏高 | 惊喜消息 |

**逐条朗读实现**：
- 每条 AI 消息气泡右侧添加朗读按钮（`R.drawable.ic_volume`）
- 用户消息气泡也可添加（可选）
- 点击后调用 `SpeechManager.playSingleMessage(text, voiceId, emotion)`
- 如果正在朗读其他内容，先停止再播放本条

**音色与角色卡绑定**：
```json
{
  "voice_profile": {
    "voice_id": "aishell3_042",
    "speed": 1.0,
    "volume": 1.0
  }
}
```

#### 2.6.1 导航架构方案（樱羽特色）

**推荐方案：保持 DrawerLayout + 按需优化**

```
┌──────────────────────────────────────────────┐
│  顶栏                                          │
│  [头像+角色名]  [设置⚙]  [新对话+]              │
├──────────────────────────────────────────────┤
│                                              │
│  消息列表区域（主内容）                          │
│  ┌──────────────────────────────────────┐    │
│  │ 用户消息（右侧气泡）                     │    │
│  │ AI 消息（左侧气泡 + 朗读按钮）            │    │
│  │ ...                                   │    │
│  └──────────────────────────────────────┘    │
│                                              │
├──────────────────────────────────────────────┤
│  输入框区域                                    │
│  [语音🎤] [文本输入________________] [发送↑]   │
└──────────────────────────────────────────────┘

左滑 → 会话抽屉（DrawerLayout）
  ├── 会话列表（RecyclerView）
  │   ├── 会话 1（当前高亮）
  │   ├── 会话 2
  │   └── ...
  ├── [新建会话]
  └── [设置] → SettingsDetailActivity
```

**设计原则**：
- 主页面 = 聊天界面（最常用功能，零层级直达）
- 会话抽屉 = 左滑（聊天记录管理）
- 设置 = 顶栏入口（配置项）
- 世界书/插件 = 设置页内子页面（独立 Activity）
- 角色卡详情 = 顶栏头像点击（底部 Sheet）

**不引入底部导航栏的原因**：
- 当前功能数量不足以支撑 4-5 个 Tab
- 底部导航栏会占用聊天界面的宝贵空间
- 微信/QQ 的底部 Tab 是因为有多个独立功能域（聊天/通讯录/发现/我），本项目核心功能只有一个：聊天
- 会话抽屉 + 顶栏设置入口已经足够简洁
