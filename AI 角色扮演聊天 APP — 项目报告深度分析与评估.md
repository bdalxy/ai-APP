## **3. 技术方案评估**

### **3.1 Chaquopy 可行性**

- **优点**：无缝集成 Python 生态，复用 v1 代码量约 60%，降低开发成本。
- **风险**：
  - 含 C 扩展的包（如 `chromadb`、`sentence-transformers`）完全不可用 → **已解决**（替换为 Embedding API + 纯 Python 计算）。
  - 调试困难：Python 异常会包装为 Java 异常，日志需额外处理。
- **建议**：P1 阶段专门编写一个"Python 调用测试用例"，覆盖 `requests` 网络、`sqlite3` 读写、`numpy` 简单运算，确保基础环境稳定。

### **3.2 向量检索替代方案**

**原方案**：DeepSeek Embedding API + 纯 Python 余弦相似度。

- **优势**：零额外依赖，实现简单。
- **隐患**：
  - 每条记忆都需调用 Embedding API（成本虽低，但增加延迟和网络依赖）。
  - 5000 条记忆全量加载到内存计算相似度，可能导致 Android 内存压力。
- **优化建议**：
  - 引入 SQLite 倒排索引作为过滤前步骤，减少需计算余弦相似度的候选集。
  - 设定检索超时（如 200ms），超时则返回最近保存的 Top-K 记忆。

### **3.3 API 成本控制**

原预估 ¥1.5~5/月，较为乐观。实际情况取决于用户活跃度。

- **敏感测试**：若用户每天聊天 50 轮 + 触发 10 次 LLM 提取 → 月成本可能接近 ¥15。
- **建议**：
  - 实现"省钱模式"：降低 LLM 提取频率、关闭联网搜索、使用更小模型。
  - APP 内显示"今日预估消耗"，用户可手动调整限额。

## **4. 架构设计点评**

### **4.1 模块依赖关系**

原设计清晰，但存在**循环依赖风险**：

- `RolePlayer` 依赖 `MemoryStorage`，`MemoryExtractor` 又依赖 `RolePlayer`。
- **建议**：在 `app_context.py` 中集中管理依赖，避免模块间直接 import 对方。

### **4.2 目录结构**

合理，但缺少以下关键目录：

- `exceptions/` – 自定义异常（API 超时、记忆提取失败等）
- `utils/` – 日志、重试装饰器、时间处理等
- `test/` – 单元测试脚本（可在 Python 环境运行，不依赖 Android）

### **4.3 v1 代码复用策略**

| 模块 | 复用率 | 复用风险 |
|------|--------|----------|
| `card_parser.py` | 95% | 角色卡格式扩展后需同步更新 |
| `deepseek.py` | 90% | 需增加重试、超时、成本计数钩子 |
| `role_player.py` | 40% | 重构幅度大，建议完全重写，仅复用 Prompt 构造逻辑 |

**建议**：为每个复用模块编写适配层，而非直接复制粘贴，便于后续升级。

## **5. 开发路线图分析**

### **当前路线**

| 阶段 | 关键验证点 | 风险 |
|------|-----------|------|
| P0 原型 | API 连通、记忆提取准确率 | 低 |
| P1 Chaquopy 验证 | 依赖兼容性 | 中 |
| P2 核心对话 | 10 轮人设不崩塌 | 低 |
| P3 记忆系统 | 准确率 >80% | 高 |
| P4 Android 封装 | APK 运行、内存不泄露 | 高 |
| P5 主动消息 | 休息时段 0 误触发 | 高 |
| P6 世界书 | 与角色卡联动 | 低 |
| P7 打磨 | 24h 稳定性 | 中 |

**总体评估**：路线合理，但 P3、P4、P5 应预留缓冲时间（各 +50% 时间预算）。

### **缺失的关键阶段**

- **用户验收测试（UAT）**：应在 P7 之前加入一个 Beta 测试阶段，招募 5~10 名真实用户试用 1 周。
- **数据迁移与备份**：未明确实现时机。建议在 P4 完成时支持对话导出为 JSON，P7 支持自动备份。

## **6. 风险评估补充**

原报告已列出 R1~R9，以下补充两项**未被充分评估的风险**：

| ID | 风险 | 概率 | 影响 | 缓解措施 |
|----|------|------|------|----------|
| R10 | DeepSeek Embedding API 返回向量维度变动 | 5% | 中 | 在代码中固定维度，若变动则触发告警并重建向量索引 |
| R11 | Android 多进程场景（WorkManager 独立进程）下 Python 全局状态丢失 | 60% | 高 | 将所有状态持久化到 SQLite，不依赖进程内全局变量 |

**特别提醒 R11**：WorkManager 可能运行在单独进程，而 Chaquopy 的 Python 环境在每个进程中独立初始化。因此 `app_context.py` 中的单例将失效。**必须在每次调用 Python 代码时从数据库重建状态**。
