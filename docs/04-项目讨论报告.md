## 1. 项目概述

### 1.1 项目定位

一个**纯本地运行的 Android AI 角色扮演聊天 APP**，单 APK 安装即用。核心目标是让 AI 角色像真人一样与用户聊天，具备记忆、主动发起话题等能力。

### 1.2 核心约束

| 约束 | 说明 |
|------|------|
| 无服务器 | 不能依赖任何服务器，纯本地运行 |
| 单 APK | 一个安装包，安装即用 |
| 低成本 | 典型月 API 成本控制在 ¥1.50 以内，上限 ¥5 |
| 好架构 | 可维护、可扩展 |
| 代码分工 | AI（Trae）全权编写，用户负责需求定义和测试验收 |

### 1.3 总体评价

> **结论：项目方案成熟、风险可控，具备进入开发阶段的条件。**
> 来源：架构师、QA 测试、后端开发、DeepSeek 四方一致认可。

## 2. 需求确认

### 2.1 三大核心体验

#### 聊天质量 — "像真人在微信上聊天"

- 多段短对话为主（模拟微信聊天风格）
- 偶有长对话
- 角色卡驱动，人设稳定不崩塌
- 对话风格自然，避免机械感
- 风险提示：人设稳定性依赖 Prompt 质量，需 P0 阶段充分测试

#### 记忆系统 — "像真人的记忆"

三层记忆架构：短期记忆（最近 50 轮对话）→ 长期记忆（普通记忆 + 强化记忆）→ 关联记忆（初期简化）

#### 主动消息 — "AI 能主动找你聊天"

- 用户设定起床/睡觉时间，划定休息时段和活跃时段
- 活跃时段按间隔随机发送
- 休息时段默认不发（除非角色卡强烈想发）
- 高风险：国产 ROM 杀后台严重，需保活方案

### 2.2 角色卡 + 世界书

角色卡：JSON 格式，定义角色性格、背景、对话风格、记忆关注点、主动消息风格等。

世界书：JSON 格式，定义世界观。默认"三次元现实世界"，用户可自行添加。

### 2.3 语音功能

- 不每句话转语音，AI 角色卡自行决定是否发语音
- TTS 文字转语音待定，暂不实现，但架构预留 TTS 扩展点

## 3. 技术方案

### 3.1 方案选择：Chaquopy

**Chaquopy** = Android 原生 APP 内嵌 Python 引擎。

选择理由：Python 写 AI 逻辑，Kotlin Compose 写 UI，单 APK 安装，MIT 许可证。

已知限制：APK 体积增加约 40-50MB，纯 Python 包完全兼容，含 C 扩展的包不可用。

### 3.2 Python 包兼容性清单

| 包名 | Chaquopy 兼容性 | 说明 |
|------|----------------|------|
| `requests` | **完全兼容** | 纯 Python，v1 核心依赖 |
| `sqlite3` | **内置支持** | Python 标准库 |
| `python-dotenv` | **完全兼容** | 纯 Python |
| `numpy` | **Chaquopy 预编译支持** | 有 Android wheel |
| `chromadb` | **不可用** | 依赖 C 扩展 |
| `sentence-transformers` | **不可用** | 依赖 PyTorch |

### 3.3 向量检索方案：DeepSeek Embedding API + 纯 Python 余弦相似度

- 存储：SQLite（JSON 字段存储向量）
- 嵌入：DeepSeek Embedding API 生成向量
- 检索：纯 Python 余弦相似度计算
- 5000 条记忆检索耗时约 50-100ms
- 优化措施：引入 SQLite 关键词倒排索引作为预过滤，设定检索超时 200ms
- 降级方案：如果 Embedding API 不可用，降级为纯关键词匹配检索

## 4. 架构设计

### 4.1 模块依赖图

```
Application (Android Kotlin)
    │
    ├── RolePlayer（对话入口）
    │   ├── ContextManager（短期记忆）
    │   ├── CardManager（角色卡）
    │   ├── WorldBookManager（世界书）
    │   ├── MemoryStorage（长期记忆检索）
    │   │   ├── VectorStore（SQLite + 向量）
    │   │   └── MemoryExtractor（规则 + LLM，异步回调）
    │   ├── PromptBuilder（组装 prompt）
    │   └── DeepSeekClient（API 调用）
    │
    └── ProactiveWorker（WorkManager 触发）
        └── ProactiveEngine（Python 决策）
```

### 4.2 循环依赖风险与解决方案

问题：`RolePlayer` 依赖 `MemoryStorage`，`MemoryExtractor` 又依赖 `RolePlayer`，存在循环依赖风险。

解决方案：在 `app_context.py` 中集中管理所有依赖，记忆提取采用异步回调模式。

### 4.3 v1 旧代码复用策略

| 模块 | 复用率 | 复用方式 |
|------|--------|----------|
| `card_parser.py` | 95% | 直接复用 |
| `card_manager.py` | 90% | 直接复用 |
| `context_manager.py` | 70% | 重构 |
| `role_player.py` | 40% | 建议完全重写 |
| `base.py` | 100% | 直接复用 |
| `deepseek.py` | 90% | 增加重试/超时/成本计数钩子 |
| `wx_adapter/` | 0% | 完全废弃 |
| `api_server.py` | 0% | 完全废弃 |

## 5. 模块详细设计

### 5.1 聊天引擎

数据流：用户输入 → ContextManager 构建上下文 → PromptBuilder 组装 System Prompt → DeepSeekClient 调用 API（含重试、超时、成本计数）→ 返回回复 → 保存到短期记忆 → 异步触发记忆提取

### 5.2 记忆系统

短期记忆：最近 50 轮对话，deque 数据结构

长期记忆：
- 普通记忆：规则提取（免费）+ LLM 提取（付费），时间衰减
- 强化记忆（Pinned）：用户说"记住"/"别忘了"/点按钮，或角色卡认定重要，衰减极慢，检索权重 ×2.0

记忆提取 Prompt：从对话中提取值得长期记住的事实，关注个人信息、偏好、重要事件、情感表达。

### 5.3 主动消息引擎

配置：wake_time、sleep_time、interval_minutes、random_factor

决策逻辑：间隔检查 → 时间窗口检查 → 随机因子（±30%）→ 起床/睡觉特殊消息

国产 ROM 保活方案：首次启动引导用户开启自启动、忽略电池优化、锁定后台，支持小米/华为/OPPO/vivo。

### 5.4 世界书

默认世界书：三次元现实世界。注入方式：按需检索而非全量注入。

### 5.5 向量存储

轻量级向量存储（纯 Python + SQLite），关键词预过滤 + 余弦相似度检索，带超时机制。

## 7. API 成本分析

### 典型用户成本（每日 20 轮对话）

| 场景 | 月成本 |
|------|--------|
| 普通聊天 | ¥1.20 |
| 记忆提取（LLM） | ¥0.10 |
| 主动消息生成 | ¥0.07 |
| Embedding API | ¥0.03 |
| **合计** | **¥1.40** |

高频用户（每天 50 轮对话 + 10 次 LLM 提取）→ 月成本可能接近 ¥15。

成本控制措施：Token 硬限制、LLM 提取节流、主动消息日限额、"省钱模式"、用户可手动调整日预算上限。

## 8. 风险评估全集

### 8.1 风险矩阵（R1-R11）

| ID | 风险 | 概率 | 影响 | 等级 | 缓解措施 |
|----|------|------|------|------|----------|
| R1 | ChromaDB 在 Chaquopy 不可用 | 100% | 高 | **已落地** | DeepSeek Embedding API + 纯 Python |
| R2 | 国产 ROM 限制后台任务 | 80% | 高 | **高** | 前台服务保活 + 厂商适配引导 |
| R3 | DeepSeek API 不稳定 | 30% | 高 | **高** | 3 次重试 + 降级回复 + 离线兜底 |
| R4 | 记忆提取质量不稳定 | 50% | 高 | **高** | 去重校验 + 矛盾检测 + 置信度字段 |
| R5 | Python→Android 性能落差 | 50% | 中 | **中** | P0 阶段做性能基准 |
| R6 | 主动消息时机不当 | 40% | 中 | **中** | 冷却期 + 用户活跃检测 + 全局开关 |
| R7 | API 成本失控 | 30% | 中 | **中** | Token 限制 + 日预算上限 + 省钱模式 |
| R8 | 上下文过长导致延迟 | 60% | 低 | **低** | 动态裁剪 + 摘要压缩 |
| R9 | API Key 泄露 | 15% | 中 | **低** | Android Keystore 加密存储 |
| R10 | Embedding API 向量维度变动 | 5% | 中 | **低** | 固定维度校验 |
| **R11** | **WorkManager 多进程下 Python 全局状态丢失** | **60%** | **高** | **高** | **所有状态持久化到 SQLite** |

### 8.2 离线兜底方案

当网络不可用或 API 失败时：返回预设回复，记录重试队列，网络恢复后自动重试。

## 10. 交叉审核过程与结论

### 审核链

用户需求 → 架构师审核 → QA 测试审核 → 后端开发审核 → 生成报告 → DeepSeek 交叉评估 → 合并最终报告

### 四方审核意见汇总

架构师：ChromaDB 不可用→替代方案，主动消息不可靠→保活，关联记忆推迟→P8，路线调整→P1加验证/P4提前。

QA 测试：需要降级回复+离线兜底，记忆提取去重+矛盾检测，API 成本日预算上限，流式输出掩盖延迟。

后端开发：向量检索用 DeepSeek Embedding API，纯 Python 余弦相似度可行，role_player.py 建议完全重写。

DeepSeek 交叉评估：R11 WorkManager 多进程状态丢失（关键新增），R10 Embedding API 维度变动，循环依赖风险，向量检索需关键词预过滤+超时，P3/P4/P5 各+50% 时间缓冲，缺少 Beta 测试阶段。

### 审核结论

> **四方一致认为：项目方案可行，风险可控，可以启动 P0 原型验证。**
