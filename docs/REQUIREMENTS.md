## 一、项目概述

AI Companion 是一款运行在 Android 手机上的 AI 角色扮演聊天应用。用户与预设的 AI 角色（"小美"）进行自然对话，AI 会记住用户说过的重要信息，下次聊天时能主动回忆。

- **目标平台**: Android 8.0+ (API 26+)
- **技术栈**: Kotlin + Chaquopy + Python + DeepSeek API + SQLite
- **部署方式**: 纯 APK 安装，无需额外运行环境
- **数据策略**: 所有数据存储在本地，仅 DeepSeek API 调用需要网络

## 三、当前阶段需求: P4 收尾 + P5 主动推送

### P4.1 手机端验证完整记忆链路 [优先级: P0]

**需求描述**:
在真机上验证 "对话→自动记忆存储→下次对话检索→注入→AI提及" 的完整链路。

**验收标准**:
1. 进行一次对话（如告诉AI"我喜欢草莓蛋糕"）
2. 开始新对话，再次提及相关话题（如"你还记得我喜欢什么吗"）
3. AI应在回复中自然提及之前存储的记忆
4. logcat 日志输出正确的记忆存储/检索/注入记录

**前置依赖**: USB连接真机

---

### P4.2 记忆可视化 [优先级: P2]

**需求描述**:
在设置面板中增加记忆管理界面，用户可以查看、编辑、删除已存储的记忆。

**具体功能**:
- 记忆列表展示（分类型、按时序）
- 单条记忆详情查看（内容、类型、重要性、衰减因子、时间戳）
- 记忆内容编辑（修改content字段）
- 单条记忆删除
- 全量记忆清空（带二次确认）
- 记忆搜索（关键词过滤）

**验收标准**:
- 列表正确显示所有记忆，可按类型筛选
- 编辑后记忆内容正确更新
- 删除后记忆从数据库移除，下次检索不再出现
- 清空操作有确认对话框
- 操作失败有错误提示

---

### P4.3 LLM 提取模式启用 + 节流 [优先级: P2]

**需求描述**:
当前仅使用规则模式（正则）提取记忆，遗漏复杂语义信息。需要引入LLM模式，但必须配合节流策略控制成本。

**具体需求**:
- 每 N 轮对话（默认5轮）触发一次 LLM 提取
- 其他轮次使用规则模式
- 用户可在设置中配置 N 的值（1-20轮）
- LLM提取使用专用的低成本模型（如 v4-flash）

**验收标准**:
- 第1轮对话: 规则模式提取
- 第5轮对话: LLM模式提取（日志可见）
- 第6-9轮: 规则模式提取
- 第10轮: 再次LLM提取
- 设置中可调整节流间隔

---

### P4.4 Embedding 缓存 (LRU) [优先级: P3]

**需求描述**:
对已调用过 embed API 的文本内容做缓存，相同文本不再重复请求。

**接口**:
- `DeepSeekClient.embed_cached(texts)` -- 先查缓存，未命中再调API
- LRU 淘汰策略，默认容量 500 条

**验收标准**:
- 相同文本二次调用 embed 时耗时<1ms(缓存命中)
- 缓存满后正确淘汰最久未使用条目
- 可通过日志区分命中/未命中

---

### P4.5 全量回退 OOM 防护 [优先级: P3]

**需求描述**:
当倒排索引无候选时，retriever会全量加载所有记忆做余弦计算。记忆量>2000条时存在OOM风险。需要增加分页加载。

**具体需求**:
- 全量回退时分页加载，每页最多 500 条
- 每页独立计算相似度，保留 top_k*3 候选
- 最终合并排序取 top_k

**验收标准**:
- 记忆量2000+时全量回退不OOM
- 检索结果与不分页版本一致

---

### P4.6 记忆归档/摘要 [优先级: P4]

**需求描述**:
长期运行后记忆量过大，需要用LLM对旧记忆做摘要压缩。

**具体需求**:
- 当记忆量超过阈值（默认500条），自动触发归档
- 选择最旧的100条同类型记忆，用LLM生成摘要
- 摘要存储为新的semantic类型记忆
- 原记忆标记为archived（不删除，但检索权重降低）

---

### P5.1 定时主动消息 [优先级: P1]

**需求描述**:
AI角色在某些时机主动发送消息，模拟真人主动联系的行为。

**具体需求**:
- 基于时间段（用户在设置中配的wake_time/sleep_time）
- 基于时间间隔（proactive_interval_minutes，默认120分钟）
- 基于情境触发（如"用户长时间未对话"）
- 使用已有的proactive引擎（decision_engine + topic_generator）

**验收标准**:
- 指定时间间隔后AI主动生成消息
- 消息内容符合角色人设和上下文
- 夜间（sleep_time后）不发送
- 用户可开关此功能

---

### P5.2 Android 系统通知 [优先级: P1]

**需求描述**:
主动消息到达时弹出Android系统通知。

**具体需求**:
- 创建NotificationChannel（"AI Companion"）
- 通知内容包含角色名和消息摘要
- 点击通知跳转回对话界面
- 前台服务保持后台存活

**验收标准**:
- 主动消息生成后系统通知栏弹出
- 点击通知正确跳转回APP
- 通知不打扰（遵守用户设置的时间段）

---

### P5.3 WorkManager 定时器 [优先级: P2]

**需求描述**:
用WorkManager替代当前CLI中的手动Schedule，实现可靠的周期性后台任务。

**具体需求**:
- PeriodicWorkRequest，周期可配置（默认2小时）
- 约束: 网络连接可用
- 执行: 调用Python proactive引擎生成消息

## 五、技术约束

1. **无云服务依赖**: 所有数据本地存储，仅DeepSeek API例外
2. **离线优先**: 核心Python环境打包进APK，无需下载
3. **仅arm64-v8a**: 不支持32位旧设备
4. **API Key 安全**: 不嵌入APK，运行时从SharedPreferences注入
5. **不新增依赖**: 优先使用已有技术栈，避免引入新三方库
