## 三、P0 紧急问题（必须立即修复）

### P0-1: 源码完全重复 —— 双副本维护

**问题描述**：

项目存在两套完全相同的 Python 源码：
- 根级副本：`f:\Trae AI\ai-APP\src\`（用于 PC 端 `main.py`）
- Android 副本：`f:\Trae AI\ai-APP\android\app\src\main\python\src\`（用于 Chaquopy）

这两个目录下的 `api_client/`、`chat_engine/`、`memory/`、`config/`、`exceptions/`、`utils/`、`proactive/` 模块代码完全相同。`chat_bridge.py` 也是两份完全相同的副本。

**实际影响**：
- 任何模块修改需要同时更新两处，极易遗漏导致 PC/Android 行为不一致
- 已在历史报告中多次出现"PC 端修复了但 Android 端忘了同步"的问题
- 两份代码会分别被打包到 APK 和 PC 环境，浪费空间

**根因**：Chaquopy 的 Python 源码需要放在 `android/app/src/main/python/` 下，而 PC 端 `main.py` 的 `sys.path` 指向的是根级 `src/`。

**优化方案**：

```
方案A（推荐）：单源 + 符号链接
  根级保留唯一源码: f:\Trae AI\ai-APP\src\
  Android 端通过 Chaquopy 的 python.srcDir 配置指向根级 src/
  在 build.gradle.kts 中添加:
    chaquopy {
        defaultConfig {
            pythonPath = listOf(file("../../../../src"))  // 指向根级 src/
        }
    }

方案B：构建时复制
  在 Gradle 构建脚本中添加 copyPythonSources task
  构建前将根级 src/ 复制到 android/app/src/main/python/src/
  优点：不依赖 Chaquopy 的 pythonPath 特性
  缺点：构建时间增加，调试时可能混淆

方案C：发布为 Python 包
  将 src/ 打包为 .whl，两边都通过 pip install 安装
  优点：最标准的做法
  缺点：开发迭代时需要频繁重新构建
```

**推荐**：**方案A（单源 + Chaquopy pythonPath）**，需验证 Chaquopy 17.0.0 是否支持 `pythonPath` 配置。

### P0-3: 记忆系统在 PC/Android 双端的调用路径不一致

**问题描述**：

| 操作 | PC 端 (main.py) | Android 端 (chat_bridge.py) |
|------|----------------|---------------------------|
| 记忆初始化 | 直接创建 VectorStore + Retriever + Extractor | 调用 init_memory() -> MemoryOrchestrator |
| 记忆检索 | main.py 直接调用 retriever.retrieve() | chat_bridge.py 调用 orchestrator.recall() |
| 记忆提取 | main.py 直接调用 extractor.extract() | chat_bridge.py 在 _auto_remember() 中调用 orchestrator.remember() |
| 记忆存储 | main.py 直接调用 vector_store.add() | 由 orchestrator 内部处理 |

PC 端 [main.py](file://f:/Trae%20AI/ai-APP/main.py) 中的 `CompanionApp` 完全绕过了 `MemoryOrchestrator`，直接操作底层组件。Android 端 [chat_bridge.py](file://f:/Trae%20AI/ai-APP/android/app/src/main/python/chat_bridge.py) 则通过 `MemoryOrchestrator` 编排。

**实际问题**：
1. `MemoryOrchestrator` 的设计初衷是统一编排入口，PC 端却完全绕过它
2. 两端对记忆提取模式的处理不同：PC 端读取 `user_settings.get("memory_extraction_mode")` 决定 mode，Android 端固定用 `"rule"` 模式
3. PC 端手动调用 `extractor.extract()` 后手动遍历 `vector_store.add()`，没有经过 orchestrator 的 embed 补充和错误隔离逻辑
4. PC 端记忆检索时传入 `min_similarity=0.3` 参数，Android 端 orchestrator 中未传递此参数

**优化方案**：

**短期（P0）**：统一 PC 端也使用 `MemoryOrchestrator`
- `CompanionApp.__init__()` 中创建 `MemoryOrchestrator`，而非手动创建 VectorStore/Retriever/Extractor
- `_process_chat()` 中记忆检索改为调用 `self.orchestrator.recall()`
- 记忆存储改为在 `chat()` 之后调用 `self.orchestrator.remember()`（或保持 `_auto_remember` 模式）

**长期（P1）**：将 PC 端 `main.py` 和 Android 端 `chat_bridge.py` 的对话流程统一为一个 `ConversationPipeline` 类。

### P0-4: DeepSeekClient 生命周期管理混乱

**问题描述**：

`DeepSeekClient` 实例被多处共享，但创建和销毁的责任不清晰：

```
Android 端:
  chat_bridge.init()
    -> 新建 DeepSeekClient (client)
    -> 传入 RolePlayer(player)
    -> init_memory():
        -> 新建 VectorStore(db)
        -> 新建 MemoryOrchestrator(store, player.client)  ← 共享 client

  问题：当用户切换预设/模型时，chat_bridge.init() 会:
    1. 关闭旧的 player.client  <-- 此时 orchestrator 还在引用它！
    2. 创建新的 client
    3. 创建新的 player
    4. 但 orchestrator 内的旧的 client 已被关闭

  同样的问题也存在于 MemoryExtractor 和 MemoryRetriever 内部对 client 的持有。
```

**根因**：缺少明确的对象所有权模型。`DeepSeekClient` 是共享资源，但被多个对象各自持有引用，没有一个"资源管理器"统一控制其生命周期。

**优化方案**：

```python
# 引入 AppContext / ServiceLocator 模式
class AppContext:
    """应用级资源容器，统一管理共享对象生命周期。"""
    
    def __init__(self):
        self._client: DeepSeekClient | None = None
        self._player: RolePlayer | None = None
        self._orchestrator: MemoryOrchestrator | None = None
    
    def initialize(self, api_key: str, preset: str, model: str = ""):
        """统一的初始化入口。"""
        self.shutdown()  # 先清理旧资源
        
        settings.DEEPSEEK_API_KEY = api_key
        
        self._client = DeepSeekClient()
        self._player = RolePlayer(self._client, preset=preset, model=model or None)
        self._player.load_card(CARD_PATH)
        
        # 后续通过 init_memory() 挂载 orchestrator
    
    def shutdown(self):
        """统一清理所有资源。"""
        if self._orchestrator:
            self._orchestrator.close()
        if self._player and self._player.client:
            self._player.client.close()
        self._client = None
        self._player = None
        self._orchestrator = None
```

### P1-2: Kotlin-Python 跨层通信不安全

**问题描述**：

[MainActivity.kt](file://f:/Trae%20AI/ai-APP/android/app/src/main/java/com/aicompanion/app/MainActivity.kt) 中使用 `extractJsonValue()` 和 `extractJsonInt()` 方法通过正则表达式从字符串中解析 JSON 值：

```kotlin
// MainActivity.kt L367-L388
private fun extractJsonValue(text: String, key: String): String? {
    try {
        val obj = org.json.JSONObject(text)
        return if (obj.has(key)) obj.getString(key) else null
    } catch (_: Exception) {}
    // 降级：Python repr(dict) 单引号格式  ← 非常脆弱！
    val singleQuote = "'$key':\\s*'([^']*)'".toRegex()
    return singleQuote.find(text)?.groupValues?.getOrNull(1)
}
```

问题：
1. 正则降级方案匹配 Python repr 的单引号格式，如果值中包含单引号就会截断
2. `extractJsonInt` 类似的正则也存在同样问题
3. 每个调用处都需要手动解析，容易遗漏错误处理

**优化方案**：

```kotlin
// BridgeClient.kt - 类型安全的桥接层
data class InitResult(val status: String, val card: CardInfo?, val message: String?)
data class ChatResult(val status: String, val reply: String?, val message: String?)
data class MemoryStats(val status: String, val total: Int, val byType: Map<String, Int>)

class BridgeClient {
    private val py: Python = Python.getInstance()
    private val module = py.getModule("chat_bridge")
    
    fun init(preset: String, model: String): InitResult {
        val result = module.callAttr("init", preset, model)
        // Chaquopy 原生 Map 转换
        val map = result.toJava(Map::class.java) as Map<String, Any>
        return parseInitResult(map)
    }
}
```

### P1-4: 记忆存储是同步阻塞操作 —— 影响对话响应

**问题描述**：

[chat_bridge.py](file://f:/Trae%20AI/ai-APP/android/app/src/main/python/chat_bridge.py) 的 `chat()` 函数（第 99-129 行）在返回 AI 回复**之后**，同步执行 `_auto_remember()`：

```python
def chat(user_input: str) -> dict:
    # ...
    reply = _player.chat(user_input)
    
    # 同步阻塞：记忆提取 + embedding API + 数据库写入
    if _orchestrator is not None and reply:
        _auto_remember(user_input, reply)
    
    return {"status": "ok", "reply": reply}
```

`_auto_remember()` 内部调用 `orchestrator.remember()`，该操作包含：
1. 规则/LLM 记忆提取
2. 逐条调用 `client.embed()` 获取向量（**网络调用**）
3. SQLite 写入

虽然在 Kotlin 端 `sendMessage()` 已经是协程（`Dispatchers.IO`），但由于 Chaquopy 的 `callAttr()` 是同步阻塞调用，且 Python GIL 的存在，记忆存储会延长 `chat()` 的整体返回时间。

**优化方案**：

方案A：将 `_auto_remember` 移到 Kotlin 端的独立协程中
```kotlin
// sendMessage() 的扩展
launch {
    val result = chat(userInput)    // 获取回复
    displayReply(result)
    
    launch(Dispatchers.IO) {
        rememberTurn(userInput, result.reply)  // 异步存储记忆
    }
}
```

方案B：在 Python 端引入线程（注意 Chaquopy 线程安全）
```python
import threading

def chat(user_input: str) -> dict:
    reply = _player.chat(user_input)
    
    if _orchestrator is not None and reply:
        threading.Thread(
            target=_auto_remember, 
            args=(user_input, reply),
            daemon=True
        ).start()
    
    return {"status": "ok", "reply": reply}
```

**推荐方案A**，因为 Kotlin 协程的生命周期管理更可控。

### P1-5: Token 预设切换需要重建全部对象

**问题描述**：

用户切换预设（如从 balanced 切到 quality）时，[MainActivity.kt](file://f:/Trae%20AI/ai-APP/android/app/src/main/java/com/aicompanion/app/MainActivity.kt) 的 `showPresetDialog()`（第 186-201 行）会：

```kotlin
lifecycleScope.launch {
    isInitialized = false
    initPython()  // 完全重新初始化
}
```

而 `initPython()` -> `chat_bridge.init()` 会：
1. 关闭旧的 `player.client`
2. 创建新的 `DeepSeekClient`
3. 创建新的 `RolePlayer`

这导致：
- 对话上下文丢失（虽然预设切换时用户期望新对话，但应该有提示）
- 记忆系统也需要重新 `init_memory()`
- 如果有正在进行的 API 请求，会被中断

**优化方案**：

为 `RolePlayer` 添加 `apply_preset()` 方法，支持运行时切换：

```python
# role_player.py
def apply_preset(self, preset: str | TokenPreset) -> None:
    """运行时切换预设，无需重建对象。"""
    if isinstance(preset, str):
        preset = get_preset(preset)
    
    self._preset = preset
    self.client.set_model(preset.model)
    self.temperature = preset.temperature
    self.max_tokens = preset.max_tokens
    self.context = ContextManager(max_tokens=preset.context_window)
    self.prompt_builder = self._create_prompt_builder()
    self.clear_context()
```

### P2-3: 配置管理缺少统一分层

**问题描述**：

当前配置分布在三个层面：
1. `Settings` 单例 — 环境变量（API Key、模型名称等）
2. `UserSettings` 单例 — JSON 文件（16项用户偏好）
3. `AppConfig` (Kotlin) — SharedPreferences（Token预设、API Key）

三者之间没有清晰的优先级和同步机制：
- API Key 同时存在于 `AppConfig`（加密SharedPreferences）和 `Settings.DEEPSEEK_API_KEY`（由 `set_api_key()` 设置）
- Token 预设同时存在于 `AppConfig` 和 `chat_bridge._current_preset`
- `UserSettings` 在 PC 端被读取（如 `memory_extraction_mode`），但在 Android 端未被使用

**优化方案**：

```python
# 统一配置优先级
# Android: Kotlin SharedPreferences > Python Settings > 默认值
# PC: 环境变量 > .env 文件 > JSON 文件 > 默认值

class ConfigProvider:
    """统一配置入口，按优先级查找。"""
    
    def __init__(self):
        self._sources: list[ConfigSource] = []
    
    def get(self, key: str) -> object:
        for source in self._sources:
            value = source.get(key)
            if value is not None:
                return value
        return DEFAULTS.get(key)
```

### P2-5: VectorStore 加密方案在 Android 上的局限性

**问题描述**：

[vector_store.py](file://f:/Trae%20AI/ai-APP/android/app/src/main/python/src/memory/vector_store.py) 的加密逻辑依赖环境变量 `MEMORY_ENCRYPTION_KEY`（第 59 行和第 74 行）。在 Android 环境中：
- 无法设置系统环境变量
- 每次 APP 重启会生成新的随机密钥（第 63-67 行的警告日志）
- 导致**重启后所有记忆无法解密**，旧记忆变成乱码

虽然在 Android 上 `cryptography` 包通常不可用（Chaquopy 限制），但 XOR 方案同样存在密钥持久化问题。

**优化方案**：
- 在 `chat_bridge.init_memory()` 中增加密钥传递参数
- Kotlin 端通过 `EncryptedSharedPreferences` 存储密钥，传递给 Python 端
- Python 端不再依赖环境变量，而是从 `set_encryption_key()` 函数获取

```python
# chat_bridge.py 新增
def set_memory_encryption_key(key: str) -> dict:
    """从 Android 端接收加密密钥并设置到环境变量。"""
    os.environ["MEMORY_ENCRYPTION_KEY"] = key
    return {"status": "ok"}
```

## 八、附加发现：代码优点

架构中值得肯定的设计决策：

1. **明确的模块分层**：`utils/ → api_client/ → chat_engine/ → chat_bridge.py` 形成清晰的依赖方向，无循环引用
2. **异常体系设计良好**：[exceptions/](file://f:/Trae%20AI/ai-APP/android/app/src/main/python/src/exceptions/) 模块中，`APIException` 和 `MemoryException` 分别作为两种异常树的基类，子类携带 `status_code` 或 `detail` 上下文
3. **@retry 装饰器设计优秀**：[retry.py](file://f:/Trae%20AI/ai-APP/android/app/src/main/python/src/utils/retry.py) 支持指数退避、jitter、可配置异常类型，在 `DeepSeekClient.chat()` 和 `embed()` 中正确使用
4. **VectorStore 线程安全**：使用 `threading.Lock` 保护 SQLite 操作，WAL 模式 + 定期 checkpoint 控制文件增长
5. **PromptBuilder 截断策略**：按优先级裁剪各部分，且在换行符处截断，考虑了实际的 token 体验
6. **记忆内容加密**：vector_store.py 中的加密/解密逻辑是后来加上的安全措施（标注 T-FIX-04），具备降级容错
7. **临时文件 + 原子替换**：`user_settings.py` 和 `exporter.py` 都使用写临时文件后再替换的策略，防止写入中断导致数据损坏
