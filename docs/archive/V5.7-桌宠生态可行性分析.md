## 一、需求概述

用户计划在AI Companion项目基本完成后，拓展为"多端AI桌宠生态"：

1. 在Android手机上集成"手机桌宠"（类似PC桌宠的悬浮窗角色，在手机桌面/任意应用上方显示）
2. 创建PC端AI桌宠应用（Windows/macOS桌面悬浮角色）
3. 两者能连接互通，形成独立生态：角色记忆共享、对话上下文同步、角色卡配置统一

核心愿景：**"同一个AI角色，手机和电脑上都能互动，彼此知道对方的存在。"**

## 二、技术可行性分析

### 2.1 Android手机桌宠

| 维度 | 分析 |
|------|------|
| **技术方案** | Android `WindowManager` 悬浮窗 + `ForegroundService` 前台服务 |
| **技术成熟度** | 高 — Android悬浮窗是成熟技术方案，市面已有多款桌宠类App |
| **项目复用性** | 极高 — 项目已有 ProactiveService 前台服务和 NotificationHelper，悬浮窗架构可在此基础上直接扩展 |
| **集成难度** | **低** — 在现有Android项目基础上增加一个悬浮窗View + 角色渲染层即可，无需新建独立App |
| **角色渲染** | 帧动画（AnimationDrawable）或 Lottie 动画起步，后续可升级为 Live2D/Spine |
| **触发逻辑** | 现有P5主动推送引擎可直接驱动桌宠"主动搭话"，无需额外开发触发系统 |
| **交互方式** | 点击角色 → 展开对话面板（复用现有对话UI）；长按 → 移动角色位置 |

**关键实现路径：**

```
ProactiveService (已有)
    ↓ 扩展
DesktopPetOverlay (新增悬浮窗View)
    ├── PetAnimationView (角色动画容器)
    ├── PetBubbleView (对话气泡)
    └── ChatPanelFragment (对话面板，复用现有)
```

### 2.2 PC端AI桌宠

| 维度 | 分析 |
|------|------|
| **方案A：Electron/Tauri** | 跨平台桌面应用，前端生态成熟（HTML/CSS/JS渲染角色），Live2D SDK支持好；缺点：体积大，Electron ~150MB |
| **方案B：Python + PyQt/tkinter** | 直接复用项目 `src/` 下所有Python代码（对话引擎、记忆系统、API客户端），零跨语言开销；缺点：UI表现力弱，Live2D支持差 |
| **方案C（推荐）：Electron壳 + Python后端** | Electron负责桌面悬浮窗和角色渲染，内置一个本地Python进程复用项目核心代码，HTTP本地通信 |
| **PC端特殊优势** | PC端可直接复用所有Python模块，相当于已有70%后端代码 |
| **角色渲染** | Live2D Web SDK（最成熟）、Spine、或简化为GIF/Lottie |

**推荐架构：**

```
Electron App
├── renderer/ (前端UI层)
│   ├── Live2D/GIF 角色渲染
│   ├── 对话气泡UI
│   └── 设置面板
├── main/ (主进程)
│   ├── 悬浮窗管理 (BrowserWindow + alwaysOnTop)
│   ├── Python进程管理 (child_process)
│   └── WebSocket客户端 (连接手机)
└── python_core/ (复用现有src/)
    ├── chat_engine/
    ├── memory/
    └── api_client/
```

### 2.3 双端互联方案

| 维度 | 分析 |
|------|------|
| **通信协议** | **WebSocket** — 轻量、双向、适合实时对话状态同步，Android和Electron都有成熟库 |
| **服务架构** | 手机App作为WebSocket Server（手机通常有固定网络环境），PC作为Client连接 |
| **同步内容** | 对话上下文、记忆检索结果、角色卡配置、主动搭话状态 |

**架构拓扑：**

```
┌─────────────────┐         WebSocket          ┌─────────────────┐
│   Android 手机   │◄────────────────────────►│    PC 桌宠       │
│  (WebSocket Srv) │                           │  (WebSocket Cli) │
│                  │                           │                  │
│  ┌────────────┐  │                           │  ┌────────────┐  │
│  │ 悬浮窗桌宠  │  │                           │  │ Electron    │  │
│  │ + 对话引擎  │  │                           │  │ + Python    │  │
│  │ + 记忆系统  │  │                           │  │ 核心代码    │  │
│  └────────────┘  │                           │  └────────────┘  │
│        ↓         │                           │        ↓         │
│  ┌────────────┐  │                           │  ┌────────────┐  │
│  │ 本地SQLite  │  │                           │  │ 本地SQLite  │  │
│  │ (主记忆库)  │  │                           │  │ (镜像副本)  │  │
│  └────────────┘  │                           │  └────────────┘  │
└─────────────────┘                           └─────────────────┘
```

**最简实现方案：**
1. 手机App启动WebSocket Server（端口可配置）
2. PC桌宠启动后扫描局域网或手动输入IP连接
3. 对话发生时，发送端通过WebSocket广播消息，接收端同步更新记忆和上下文
4. 两端共享同一份角色卡配置（通过同步或手动导入）

**云端方案（阶段5）：**
- 部署轻量中继服务器（如Cloudflare Workers / 轻量VPS）
- 解决两端不在同一局域网时的通信问题
- 可作为共享记忆的云端备份节点

## 三、Token消耗分析

### 3.1 当前基准

- **单次对话消耗**：~500 tokens（一次完整回复 + 上下文）
- **记忆检索Embedding**：已通过LRU缓存优化，重复检索不消耗API
- **LLM提取**：现有节流机制，每5轮对话触发一次记忆提取

### 3.2 桌宠场景预估

| 场景 | 计算 | 日消耗 | 月消耗 |
|------|------|--------|--------|
| 单端（当前） | 每天10次对话 x 500 tokens | ~5,000 tokens | ~150,000 tokens |
| 单端桌宠 | 每小时2次主动搭话 x 16小时 x 500 tokens | ~16,000 tokens | ~480,000 tokens |
| 双端独立运行 | 手机 + PC 各自独立对话 | ~32,000 tokens | ~960,000 tokens |
| 双端共享上下文（优化后） | 共享上下文后减少30%重复 | ~22,400 tokens | ~672,000 tokens |

### 3.3 优化策略

| 策略 | 说明 | 预期效果 |
|------|------|----------|
| **共享对话上下文** | 双端共用同一份对话历史和记忆，避免重复检索和上下文重建 | 减少30-40% token消耗 |
| **主动搭话节流** | 桌宠主动搭话频率可配置（默认每小时1-3次），用户可设上限 | 可控消耗增长 |
| **记忆共享继承** | 记忆系统已有LLM提取节流（每5轮一次），桌宠继承此逻辑 | 无额外记忆提取开销 |
| **Embedding缓存** | 现有LRU缓存机制，双端可共享Embedding结果 | 减少重复API调用 |
| **静默时段** | 用户可设置"勿扰模式"时间段，桌宠不主动搭话 | 进一步控制消耗 |

### 3.4 结论

正常使用场景下（双端共享上下文 + 节流控制），月消耗相比单端增加约**30-50%**，在可接受范围。

以 DeepSeek 当前 API 定价计算，双端桌宠场景月消耗成本约 **人民币5-15元**（取决于主动搭话频率和对话长度），远低于同类商用AI桌宠产品的订阅费用。

## 五、风险与注意事项

| 风险项 | 等级 | 说明 | 缓解措施 |
|--------|------|------|----------|
| **悬浮窗权限** | 高 | 小米、华为、OPPO等厂商对悬浮窗权限管理策略不一致，部分需用户手动开启 | 提供权限引导页面，检测并跳转对应设置页 |
| **电池消耗** | 中 | 前台服务 + WebSocket持续运行，对手机电池有影响 | 优化轮询间隔，使用系统JobScheduler替代持续轮询 |
| **Token成本失控** | 中 | 如果主动搭话过于频繁且无上限，月消耗可能超过预期 | 硬编码频率上限，用户可配置阈值，超出后自动降级 |
| **同步延迟** | 低 | WebSocket在弱网/切换网络环境下可能断连 | 实现自动重连 + 离线消息队列 + 状态同步补偿 |
| **版权风险** | 中 | 角色素材（立绘、动画）若使用他人作品可能侵权 | 使用原创素材或开源授权资源（如CC0），Live2D模型需自行购买或制作 |
| **Google Play政策** | 低 | 悬浮窗类App在Play Store上架需符合特定政策 | 遵循Android前台服务合规要求，提供合理的权限说明 |

## 六、总结

**桌宠生态是AI Companion项目的自然延伸**。项目已有的架构积累（主动推送引擎、记忆系统、对话引擎、API客户端）为双端桌宠提供了坚实的代码基础。技术上不存在重大障碍。
