## T-FIX-01：API Key 改用 EncryptedSharedPreferences

| 属性 | 值 |
|:-----|:-----|
| **严重程度** | HIGH |
| **来源** | V5.2审查 S1 |
| **文件** | `AppConfig.kt`, `build.gradle.kts` |
| **预计工时** | 2h |

### 问题
API Key 以明文 XML 存储在 `/data/data/.../shared_prefs/ai_companion_prefs.xml`。
Root 设备或物理攻击可直接读取。

### 修复方案
1. `build.gradle.kts` 添加 `androidx.security:security-crypto` 依赖
2. `AppConfig.kt` 中 `SharedPreferences` 替换为 `EncryptedSharedPreferences`
3. 添加 `masterKey` 管理

### 测试合格标准
- APP 首次启动 → 设置 API Key → 关闭 APP → 检查 shared_prefs XML 文件内容是否加密
- 重启 APP → API Key 可正常读取使用

## T-FIX-02：set_api_key() 后更新 session header

| 属性 | 值 |
|:-----|:-----|
| **严重程度** | HIGH |
| **来源** | V5.2审查 B1 |
| **文件** | `deepseek.py`, `chat_bridge.py` |
| **预计工时** | 0.5h |

### 问题
`set_api_key()` 修改了 `settings.DEEPSEEK_API_KEY`，但已存在的 `DeepSeekClient` 实例的 `session.headers["Authorization"]` 不会更新。必须在 Python 端调用 `set_api_key()` 后重新 `init()`。

### 修复方案
`DeepSeekClient` 新增 `update_api_key(key)` 方法，同时更新 `self.session.headers`。
`chat_bridge.set_api_key()` 调用此方法更新已存在 client 的 header。

### 测试合格标准
1. 启动 APP → 发送消息（正常回复）
2. 修改 API Key → 发送消息 → 使用新 Key（正常回复，不报 401）

## T-FIX-03：Release 构建设为 WARNING 日志级别

| 属性 | 值 |
|:-----|:-----|
| **严重程度** | MEDIUM |
| **来源** | V5.2审查 S3 |
| **文件** | `settings.py`, `MainActivity.kt` |
| **预计工时** | 0.5h |

### 问题
logcat 中 INFO 级别日志泄露用户消息全文（`extractor.py` L206）和提取的记忆内容（L212）。Release 构建中仍需控制。

### 修复方案
1. Python 侧：`settings.py` 中根据构建类型设置日志级别（Debug=DEBUG, Release=WARNING）
2. Kotlin 侧：`MainActivity.kt` 中 `Log.d/i` 包含用户内容的地方添加 `BuildConfig.DEBUG` 守卫

### 测试合格标准
- Debug 构建：logcat 正常输出详细日志
- Release 构建：logcat 中无用户消息内容泄露
