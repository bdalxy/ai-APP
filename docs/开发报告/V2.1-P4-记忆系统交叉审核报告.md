## 二、P0 缺陷清单（必须修复）

### 1. delete/update TOCTOU 竞态
- **文件**：`vector_store.py` — `delete()` / `update()`
- **问题**：先 `get()` (有锁→释放锁) → 再 `execute()` (重新获取锁)，中间存在时间窗口
- **影响**：并发场景下数据一致性问题
- **修复**：将 get+write 合并入同一个 `with self._lock` 块

### 2. close() 无防护标志
- **文件**：`vector_store.py`
- **问题**：close() 后无 `_closed` 标志，后续操作直接触发 `sqlite3.ProgrammingError`
- **影响**：崩溃而非友好的错误提示
- **修复**：添加 `self._closed` 标志，在所有公开方法入口检查

### 3. get_weight() 忽略 entry.decay_factor
- **文件**：`decay.py` — `get_weight()`
- **问题**：每次都重新计算衰减，`entry.decay_factor` 字段形同虚设
- **影响**：浪费 CPU，对数万条记忆做重排时影响性能
- **修复**：优先使用 `entry.decay_factor`，仅在未设置时重新计算

### 4. 余弦相似度双重计算
- **文件**：`retriever.py` `retrieve()` + `vector_store.py` `search()`
- **问题**：VectorStore.search() 已算过余弦相似度，Retriever 又全部重算一遍
- **影响**：每次检索多做一倍向量点积运算
- **修复**：search() 直接返回 `list[tuple[MemoryEntry, float]]` 带分数

### 5. 去重基于 bigram 而非语义向量
- **文件**：`extractor.py` — `_deduplicate()`
- **问题**："我住在北京" 和 "我家在北京" 的 bigram Jaccard = 0，但语义完全相同
- **影响**：语义重复记忆无法去重，数据库膨胀
- **修复**：优先使用 embedding 余弦相似度去重，无 embedding 时回退 bigram

### 6. 错误静默吞没
- **文件**：`vector_store.py` — `_get_by_ids()` / `_get_all()` / `count()`
- **问题**：数据库错误时返回空列表/0，调用方无法区分"无数据"和"数据损坏"
- **影响**：可观测性问题，掩盖真实故障
- **修复**：抛出 `MemoryStorageError` 或至少 warning 级别日志

### 7. WAL 无 checkpoint 策略
- **文件**：`vector_store.py`
- **问题**：WAL 启用了但没有 checkpoint 调用，wal 文件会无限增长
- **影响**：长期运行可能耗尽存储空间
- **修复**：定期或在 add() 时触发 `PRAGMA wal_checkpoint(TRUNCATE)`

## 三、P1 缺陷清单（应该修复）

| # | 问题 | 文件 |
|---|------|------|
| 8 | embed API 无缓存 — 每次检索都调用，增加 200-500ms 延迟 | `retriever.py` |
| 9 | LLM 提取无节流策略 — 每轮对话都调 API，成本翻倍 | `extractor.py` |
| 10 | search() 全量回退时一次性加载所有 embedding → OOM 风险 | `vector_store.py` |
| 11 | WAL 模式在部分 Android 设备可能不可用 | `vector_store.py` |
| 12 | `_load_index()` 游标泄漏风险 | `vector_store.py` |
| 13 | 去重 O(n*m) 复杂度（n=5000 条已有记忆） | `extractor.py` |
