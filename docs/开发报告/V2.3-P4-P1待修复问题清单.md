**策略**：P0 已修复，P1 问题当前阶段（记忆量少、提取未接入）基本无影响，延后到记忆量上来前修复

## P1-1：embed API 无缓存

| 项目 | 内容 |
|------|------|
| **位置** | `retriever.py` — `retrieve()` L43-48 |
| **现象** | 每次用户发消息触发记忆检索时，都调用一次 DeepSeek Embedding API |
| **影响** | 每次消息额外增加 200-500ms 延迟；高频对话可能触发 API 限流（60 RPM） |
| **触发条件** | 记忆检索接入对话流程后，每条消息 → 1 次 embed 调用 |
| **修复方案** | 实现 LRU 缓存（最近 50 条 query embedding），同 query 或高度相似 query 跳过 API 调用 |
| **触发修复时机** | 记忆功能首次可用、日均消息 > 50 条时 |

---

## P1-2：LLM 提取无节流策略

| 项目 | 内容 |
|------|------|
| **位置** | `extractor.py` — `_extract_by_llm()` + 未来 Orchestrator 调用 |
| **现象** | `auto` 模式下每轮对话都会调用一次 DeepSeek Chat API 做记忆提取 |
| **影响** | 每次对话额外消耗 API 费用（约等同一轮对话的成本），高频场景下显著增加月费 |
| **触发条件** | Orchestrator 接入对话后，每轮对话 → 1 次额外 chat API 调用 |
| **修复方案** | 加入节流策略：每 N 轮（如 5 轮）触发一次 LLM 提取，或仅当对话内容显著变化时触发（基于文本相似度阈值） |
| **触发修复时机** | Orchestrator 开发完成后、LLM 提取模式启用时 |

---

## P1-3：search() 全量回退 OOM 风险

| 项目 | 内容 |
|------|------|
| **位置** | `vector_store.py` — `search()` L308-310 |
| **现象** | 倒排索引无命中时调用 `_get_all()`，将全部记忆（含 embedding）加载到内存。10000 条记忆 × 1536 维 ≈ 122MB |
| **影响** | 低端 Android 设备（2-4GB RAM）可能触发 OOM |
| **触发条件** | 用户查询使用了倒排索引中不存在的新关键词 → 回退全量扫描 |
| **修复方案** | 方案A：使用 SQLite 游标逐行计算相似度而非一次性加载；方案B：新增"不回退"模式，倒排无命中直接返回空 |
| **触发修复时机** | 记忆量 > 2000 条时（当前几乎不触发） |

---

## P1-4：WAL 模式在部分 Android 设备可能不可用

| 项目 | 内容 |
|------|------|
| **位置** | `vector_store.py` — `__init__()` L178 |
| **现象** | `PRAGMA journal_mode=WAL` 在部分厂商 ROM（华为部分机型、Android 10-）上可能失败 |
| **影响** | 数据库无法打开（`SQLITE_CANTOPEN`），记忆功能完全不可用 |
| **触发条件** | 使用特定 Android 设备的特定 ROM |
| **修复方案** | try/except 捕获 `sqlite3.OperationalError`，降级为 DELETE 模式 |
| **触发修复时机** | 收到用户反馈 WAL 错误时 |

---

## P1-5：`_load_index()` 游标泄漏风险

| 项目 | 内容 |
|------|------|
| **位置** | `vector_store.py` — `_load_index()` L200-203 |
| **现象** | `cursor = self._conn.execute(...)` 返回的游标在异常时不会自动关闭（Chaquopy 环境 GC 时机不确定） |
| **影响** | 极低风险，仅在启动时执行一次，且 CPython GC 通常会清理 |
| **触发条件** | `_load_index()` 过程中发生异常 |
| **修复方案** | 使用 `with self._conn as conn: cursor = conn.execute(...)` 上下文管理器 |
| **触发修复时机** | 出现启动异常日志时 |

---

## P1-6：去重 O(n*m) 复杂度

| 项目 | 内容 |
|------|------|
| **位置** | `extractor.py` — `_deduplicate()` L360-381 |
| **现象** | 每次提取都对所有已有记忆逐条比较（已按类型分组优化）。n=5000 时，每条新记忆需约 4 次 bigram 集合运算 × 5000 = 20000 次 |
| **影响** | 5000 条记忆下去重延迟约 100-500ms |
| **触发条件** | 记忆量 > 5000 且使用 bigram 去重时 |
| **修复方案** | 方案A：对已有记忆预计算 bigram 集合并缓存；方案B：batch 去重而非逐条 |
| **触发修复时机** | 记忆量 > 5000 条时（当前可忽略） |

## 修复优先级汇总

| 时机 | 问题 | 触发条件 |
|------|------|----------|
| 🔴 记忆功能可用时 | P1-1 embed 缓存 | 每条消息触发检索 |
| 🟡 LLM 提取启用时 | P1-2 提取节流 | LLM 模式启用 |
| 🟡 记忆量 > 2000 | P1-3 全量回退 OOM | 倒排索引无命中 |
| 🟢 收到反馈时 | P1-4 WAL 兼容性 | 特定设备 ROM |
| 🟢 出现异常时 | P1-5 游标泄漏 | 启动异常 |
| 🟢 记忆量 > 5000 | P1-6 去重复杂度 | bigram 去重 |

## 当前阶段影响评估

当前阶段（P4 开发中、记忆功能尚未接入对话流程），**P1 问题全部无实际影响**：
- 记忆检索未接入 → 无 embed 调用 → P1-1/P1-3 不触发
- 提取未接入 → 无 LLM 调用 → P1-2 不触发
- 记忆量 < 100 → P1-3/P1-6 无性能问题
- P1-4/P1-5 为极低概率的边缘情况
