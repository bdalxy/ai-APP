## 1. 项目概述

### 1.1 项目定位

一个**纯本地运行的 Android AI 角色扮演聊天 APP**，单 APK 安装即用。核心目标是让 AI 角色像真人一样与用户聊天，具备记忆、主动发起话题等能力。

### 1.2 核心约束

| 约束 | 说明 |
|------|------|
| 无服务器 | 不能依赖任何服务器，纯本地运行 |
| 单 APK | 一个安装包，安装即用 |
| 低成本 | 典型月成本 ¥1-3，上限 ¥5 |
| 好架构 | 可维护、可扩展 |
| 初始 APK | 目标 < 80MB（含 Python 运行时），后续按需扩展 |
| 代码分工 | AI（Trae）全权编写，用户负责需求定义和测试验收 |

### 1.3 核心结论

> **四方审核一致通过：项目方案成熟、风险可控，具备进入开发阶段的条件。**

## 2. 需求确认

### 2.1 三大核心体验

#### 聊天质量 — "像真人在微信上聊天"
- 多段短对话为主，偶有长对话
- 角色卡驱动，人设稳定不崩塌
- 对话风格自然，避免机械感

#### 记忆系统 — "像真人的记忆"

短期记忆：最近 N 轮对话（可配置，默认 20 条）→ 长期记忆：普通记忆（规则+LLM）+ 强化记忆（用户/AI标记，几乎不衰减，检索权重×2）→ 关联记忆（初期简化）

关键约束：用户有大量聊天需求，记忆量可达 K 级（数千条），需要倒排索引 + 关键词预过滤 + 定期归档。

#### 主动消息 — "AI 能主动找你聊天"
- 用户设定起床/睡觉时间，划定休息时段和活跃时段
- 活跃时段按间隔随机发送
- 休息时段默认不发（除非角色卡强烈想发）

### 2.2 角色卡 + 世界书

角色卡：JSON 格式。世界书：完全参考 Tavo APP 的格式，支持正则表达式触发。

### 2.3 设置系统（16 项）

| 分类 | 设置项 | 默认值 |
|------|--------|--------|
| 作息 | 起床/睡觉时间 | 08:00 / 23:00 |
| 主动消息 | 开关/发送间隔 | 关闭 / 120分钟 |
| 记忆 | 短期/长期记忆数量 | 20 / 500 |
| 预算 | 日预算上限 | 0（不限制） |
| 省钱 | 省钱模式 | 关闭 |
| 搜索 | 联网搜索开关 | 关闭 |
| 导出 | 自动导出/格式 | 关闭 / JSON |
| 设备 | 设备配置档位 | 自动 |

### 2.4 首次启动引导（3 步）

1. 权限说明 — 明确列出需要的权限和用途
2. 设备检测 — 检测品牌、Android 版本、内存、存储
3. 推荐配置 — 根据设备自动推荐设置

### 2.5 记忆导出

流式导出（避免 OOM），可选加密导出，存储在 Android Downloads 目录。

## 3. 技术方案

### 3.1 方案选择：Chaquopy

Kotlin/Compose UI → Chaquopy 桥接 → Python 引擎（chat_engine/memory/api_client/proactive/world_book）→ Android 基础层（SQLite、WorkManager）

### 3.2 Python 包兼容性

| 包名 | Chaquopy 兼容性 | 说明 |
|------|----------------|------|
| `requests` | 完全兼容 | 纯 Python |
| `sqlite3` | 内置支持 | Python 标准库 |
| `python-dotenv` | 完全兼容 | 纯 Python |
| `loguru` | 完全兼容 | 纯 Python，日志 |
| `numpy` | Chaquopy 预编译支持 | 可选，K 级记忆时引入 |
| `chromadb` | 不可用 | 放弃 |
| `sentence-transformers` | 不可用 | 放弃 |

### 3.3 向量检索方案（针对 K 级记忆）

核心方案：DeepSeek Embedding API + 倒排索引预过滤 + 纯 Python 余弦相似度

流程：用户消息 → 关键词提取(bigram) → 倒排索引预过滤(50ms) → Top-500候选集 → 余弦相似度精排(50ms) → Top-K结果

性能预估：

| 数据量 | 全量扫描耗时 | 索引预过滤后耗时 |
|--------|-------------|-----------------|
| 5,000 | ~500ms | ~50ms |
| 10,000 | ~1,000ms | ~80ms |
| 20,000 | ~2,000ms | ~120ms |

定期归档：超过 30 天未使用的记忆自动归档。

### 3.4 代码复用策略（最终决定）

> **全部重写，旧代码仅作参考。** 架构差异过大，强行复用得不偿失。

### 3.5 APK 精简策略

- 初始 APK 目标 < 80MB，实际预估约 35-40MB（release）
- 核心依赖仅 3 个：`requests`、`python-dotenv`、`loguru`
- 初期不引入 `numpy`，K 级记忆时按需引入
- 单 ABI（arm64-v8a），不打包多架构

## 4. 架构设计

### 4.1 模块依赖图

```
Application (Android Kotlin)
    │
    ├── RolePlayer（对话入口）
    │   ├── ContextManager（短期记忆）
    │   ├── CardManager（角色卡）
    │   ├── WorldBookEngine（世界书，正则匹配）
    │   ├── MemoryRetriever（长期记忆检索）
    │   │   ├── VectorStore（倒排索引 + 余弦相似度）
    │   │   └── MemoryExtractor（规则 + LLM，异步回调）
    │   ├── PromptBuilder（组装 prompt）
    │   └── DeepSeekClient（API 调用）
    │
    ├── ProactiveWorker（WorkManager 触发）
    │   └── ProactiveEngine（Python 决策）
    │
    ├── SettingsManager（设置管理，观察者模式）
    │   └── 热更新通知 → 各模块监听器
    │
    └── MemoryExporter（导出引擎）
        ├── 流式导出聊天记录
        └── 流式导出长期记忆
```

## 5. 模块详细设计

### 5.1 聊天引擎

数据流：用户输入 → ContextManager 构建上下文 → PromptBuilder 组装 System Prompt → DeepSeekClient 调用 API（含重试、超时、成本计数）→ 返回回复 → 保存到短期记忆 → 异步触发记忆提取

### 5.2 记忆系统（K 级优化）

倒排索引 + 向量检索：关键词提取(bigram) → 倒排索引预过滤 → 余弦相似度精排 → 时间衰减 + 强化记忆加权

定期归档：超过 30 天未使用的记忆归档到 archive_*.json，活跃记忆总量控制在 max_active 以内。

### 5.3 世界书（Tavo 兼容）

支持正则表达式触发，包含 keys、regex_keys、content、constant、probability、priority、max_injections、cooldown_rounds 等字段。

正则匹配引擎：constant 始终注入 → 正则匹配 → 关键词匹配 → 概率检查 → 冷却检查。

正则安全（ReDoS 防护）：静态检测危险模式 + 动态超时检测。

### 5.4 设置系统（16 项 + 热更新）

存储方案：Python 侧 JSON 文件，Kotlin 侧 DataStore 作为主副本。

观察者模式（热更新）：SettingsManager 通知所有监听器，记忆模块、主动消息模块等实时响应设置变更。

设备预设（3 档）：low（<4GB RAM）、medium（4-6GB）、high（>6GB）。

### 5.5 记忆导出

流式导出（避免 OOM）：分批读取，逐批写入 JSON 文件。加密导出预留接口。

### 5.6 首次启动引导

3 步流程：权限说明 → 设备检测 → 推荐配置（一键应用或手动调整）。

## 6. API 成本分析（基于官方定价）

### 6.1 DeepSeek 最新官方定价

来源：[DeepSeek 官方定价页](https://api-docs.deepseek.com/zh-cn/quick_start/pricing/) — 2026 年 6 月

| 模型 | 缓存命中输入 | 缓存未命中输入 | 输出 |
|------|-------------|--------------|------|
| **deepseek-v4-flash** | ¥0.02/百万 | ¥1/百万 | ¥2/百万 |
| **deepseek-v4-pro** | ¥0.025/百万 | ¥3/百万 | ¥6/百万 |

模型名称变更：`deepseek-chat` → `deepseek-v4-flash`

### 6.2 典型用户月度成本（50 轮/天）

| 场景 | Flash 模式 | Pro 模式 |
|------|-----------|---------|
| 50 轮/天 | **¥1.07** | ¥3.18 |
| 200 轮/天（重度） | ¥4.28 | ¥12.70 |

结论：即使是重度用户 + Pro 模型，月成本不到 ¥15。

### 6.3 成本控制措施

1. Token 硬限制：单次请求上下文 ≤ 4000 token
2. LLM 提取节流：每 5 轮对话触发一次
3. 主动消息日限额：每天最多 10 条
4. 世界书按需检索
5. 本地成本计数器，APP 内显示预估费用
6. 省钱模式：降低 LLM 提取频率、关闭联网搜索
7. 用户可手动调整日预算上限

## 7. 风险评估全集

### 7.1 风险矩阵（R1-R15，完整合并）

| ID | 风险 | 概率 | 影响 | 等级 | 缓解措施 |
|----|------|------|------|------|----------|
| R1 | ChromaDB 不可用 | 100% | 高 | 已落地 | DeepSeek Embedding API + 纯 Python |
| R2 | 国产 ROM 限制后台 | 80% | 高 | 高 | 前台服务保活 + 厂商适配引导 |
| R3 | DeepSeek API 不稳定 | 30% | 高 | 高 | 3 次重试 + 降级回复 + 离线兜底 |
| R4 | 记忆提取质量不稳定 | 50% | 高 | 高 | 去重 + 矛盾检测 + 置信度 |
| R5 | K 级记忆检索性能 | 60% | 高 | 高 | 倒排索引预过滤 + 定期归档 |
| R6 | 主动消息时机不当 | 40% | 中 | 中 | 冷却期 + 用户活跃检测 + 全局开关 |
| R7 | API 成本失控 | 30% | 中 | 中 | Token 限制 + 日预算 + 省钱模式 |
| R8 | 上下文过长 | 60% | 低 | 低 | 动态裁剪 + 摘要压缩 |
| R9 | API Key 泄露 | 15% | 中 | 低 | Android Keystore 加密 |
| R10 | Embedding API 维度变动 | 5% | 中 | 低 | 固定维度校验 |
| R11 | WorkManager 多进程状态丢失 | 60% | 高 | 高 | 所有状态持久化到 SQLite |
| R12 | 世界书正则 ReDoS | 30% | 高 | 高 | 静态检测 + 动态超时 |
| R13 | 导出 OOM | 50% | 高 | 高 | 流式导出，分批读写 |
| R14 | 设置热更新失效 | 40% | 中 | 中 | 观察者模式 + 监听器注册 |
| R15 | 设备检测不准确 | 30% | 中 | 中 | 悲观策略，推荐低一档 |

### 7.2 离线兜底方案

当网络不可用或 API 失败时：返回预设回复，记录重试队列，网络恢复后自动重试，离线时主动消息功能暂停。

### 7.3 关键边界条件

空状态、极限状态（对话 500+ 轮、记忆 20,000 条）、异常输入（纯表情、Unicode、注入攻击、ReDoS）、状态切换（WiFi/4G、息屏/亮屏、APP 后台/前台、手机重启）、并发（用户快速发送 + 主动消息同时到达）。
