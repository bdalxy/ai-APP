## 1. 项目概述

### 1.1 项目定位

一个**纯本地运行的 Android AI 角色扮演聊天 APP**，单 APK 安装即用。核心目标是让 AI 角色像真人一样与用户聊天，具备记忆、主动发起话题等能力。

### 1.2 核心约束

| 约束 | 说明 |
|------|------|
| 无服务器 | 不能依赖任何服务器，纯本地运行 |
| 单 APK | 一个安装包，安装即用 |
| 低成本 | 典型月 API 成本控制在 ¥1.50 以内，上限 ¥5 |
| 好架构 | 可维护、可扩展 |
| 代码分工 | AI（Trae）全权编写，用户负责需求定义和测试验收 |

### 1.3 总体评价

> **结论：项目方案成熟、风险可控，具备进入开发阶段的条件。**
>
> 来源：架构师、QA 测试、后端开发、DeepSeek 四方一致认可。

## 2. 需求确认

### 2.1 三大核心体验

#### 聊天质量 — "像真人在微信上聊天"

- 多段短对话为主（模拟微信聊天风格）
- 偶有长对话
- 角色卡驱动，人设稳定不崩塌
- 对话风格自然，避免机械感
- ⚠️ **风险提示**：人设稳定性依赖 Prompt 质量，需 P0 阶段充分测试

#### 记忆系统 — "像真人的记忆"

三层记忆架构：

```
短期记忆：最近 50 轮对话，保证对话连贯性
    ↓ 定期筛选
长期记忆：
  ├─ 普通记忆：规则提取 + LLM 提取，时间衰减
  └─ 强化记忆：用户/AI 标记，几乎不衰减，检索权重 ×2
    ↓ 关联分析
关联记忆：记忆之间的关联关系（初期简化为同 session 关联）
```

#### 主动消息 — "AI 能主动找你聊天"

- 用户设定起床/睡觉时间，划定休息时段和活跃时段
- 活跃时段按间隔随机发送
- 休息时段默认不发（除非角色卡强烈想发）
- 起床/睡觉时有特殊问候消息
- ⚠️ **高风险**：国产 ROM 杀后台严重，需保活方案（详见 §8.1 R2）

### 2.2 角色卡 + 世界书

**角色卡**：JSON 格式，定义角色性格、背景、对话风格、记忆关注点、主动消息风格等。

**世界书**：JSON 格式，定义世界观。默认"三次元现实世界"，用户可自行添加。

### 2.3 语音功能

- 不每句话转语音，AI 角色卡自行决定是否发语音
- TTS 文字转语音待定，暂不实现，但架构预留 TTS 扩展点

## 3. 技术方案

### 3.1 方案选择：Chaquopy

**Chaquopy** = Android 原生 APP 内嵌 Python 引擎。

```
┌──────────────────────────────────────┐
│            Android APK               │
│                                      │
│  Kotlin/Compose UI（原生界面）         │
│         ↓ 调用                        │
│  Chaquopy Python 引擎                 │
│    ├── chat_engine/  角色扮演         │
│    ├── memory/       记忆系统         │
│    ├── api_client/   API 调用         │
│    ├── proactive/    主动消息         │
│    └── world_book/   世界书           │
│         ↓                            │
│  Android 基础层（SQLite、WorkManager） │
└──────────────────────────────────────┘
```

**选择理由**：
- Python 写 AI 逻辑，复用 v1 约 60% 代码
- Kotlin Compose 写 UI，原生体验
- 单 APK 安装，不需要 Termux 或服务器
- Chaquopy MIT 许可证，免费开源

**已知限制**：
- APK 体积增加约 40-50MB（含 Python 运行时）
- 纯 Python 包完全兼容，含 C 扩展的包不可用
- Python 异常会包装为 Java 异常，调试需额外日志处理
- **P1 阶段必须编写"Python 调用测试用例"**，覆盖 `requests`、`sqlite3`、`numpy`

### 3.2 Python 包兼容性清单

| 包名 | Chaquopy 兼容性 | 说明 |
|------|----------------|------|
| `requests` | **完全兼容** | 纯 Python，v1 核心依赖 |
| `sqlite3` | **内置支持** | Python 标准库 |
| `json` | **内置支持** | Python 标准库 |
| `python-dotenv` | **完全兼容** | 纯 Python |
| `numpy` | **Chaquopy 预编译支持** | 有 Android wheel，约 4.8MB |
| `scikit-learn` | **Chaquopy 预编译支持** | 有 Android wheel，约 8.4MB |
| `chromadb` | **不可用** | 依赖 onnxruntime、hnswlib 等 C 扩展 |
| `sentence-transformers` | **不可用** | 依赖 PyTorch |
| `faiss-cpu` | **不可用** | 无 Android wheel |

### 3.3 向量检索方案：DeepSeek Embedding API + 纯 Python 余弦相似度

**方案**：
- 存储：SQLite（JSON 字段存储向量）
- 嵌入：DeepSeek Embedding API 生成向量
- 检索：纯 Python 余弦相似度计算

**性能分析**：
- 5000 条记忆检索耗时约 50-100ms（可接受）
- 5000 条向量内存占用约 20MB（可接受）
- 零额外依赖，不增加 APK 体积

**优化措施**（DeepSeek 建议）：
- 引入 SQLite 关键词倒排索引作为预过滤，减少候选集
- 设定检索超时 200ms，超时返回最近保存的 Top-K
- 记忆量超过 10000 条时引入 numpy 加速

**降级方案**：如果 Embedding API 不可用，降级为纯关键词匹配检索

### 4.1 模块依赖图

```
Application (Android Kotlin)
    │
    ├── RolePlayer（对话入口）
    │   ├── ContextManager（短期记忆）
    │   ├── CardManager（角色卡）
    │   ├── WorldBookManager（世界书）
    │   ├── MemoryStorage（长期记忆检索）
    │   │   ├── VectorStore（SQLite + 向量）
    │   │   └── MemoryExtractor（规则 + LLM，异步回调）
    │   ├── PromptBuilder（组装 prompt）
    │   └── DeepSeekClient（API 调用）
    │
    └── ProactiveWorker（WorkManager 触发）
        └── ProactiveEngine（Python 决策）
            ├── TopicGenerator（话题生成）
            └── DeepSeekClient（API 调用）
```

### 4.2 循环依赖风险与解决方案

**问题**（DeepSeek 发现）：`RolePlayer` 依赖 `MemoryStorage`，`MemoryExtractor` 又依赖 `RolePlayer`，存在循环依赖风险。

**解决方案**：
- 在 `app_context.py` 中集中管理所有依赖，避免模块间直接 import
- 记忆提取采用**异步回调模式**：`RolePlayer` 完成对话后，通过事件通知触发提取，而非直接调用
- 提取器不依赖 `RolePlayer`，只依赖 `DeepSeekClient` 和 `VectorStore`

### 5.2 记忆系统

#### 短期记忆
- 最近 50 轮对话，`deque` 数据结构
- 每轮对话后自动更新，直接拼入上下文

#### 长期记忆

**普通记忆**：
- 规则提取（免费）：匹配固定句式
- LLM 提取（付费）：重要对话（>20 轮或用户标记）才触发
- 时间衰减：`score = importance × e^(-λ × days)`，λ=0.01

**强化记忆（Pinned）**：
- 触发：用户说"记住"/"别忘了"/点 📌 按钮，或角色卡认定重要
- 衰减系数 λ=0.001（极慢）
- 检索时权重 ×2.0

#### 记忆提取 Prompt

```

【角色设定】
{character_name}：{character_personality}

【对话内容】
{conversation_text}

【提取规则】
1. 只提取对角色有意义的、值得长期记住的事实
2. 重点关注：用户的个人信息、偏好、重要事件、情感表达
3. 如果对话是日常寒暄，返回空列表

【强化记忆判断】
如果对话中出现以下情况，标记 is_pinned=true：
- 用户明确说"记住"、"别忘了"、"很重要"

### 5.3 主动消息引擎

#### 配置

```json
{
  "proactive": {
    "wake_time": "08:00",
    "sleep_time": "23:00",
    "interval_minutes": 120,
    "random_factor": 0.3
  }
}
```

#### 决策逻辑

```python
def should_send_message(now, config, character_card, last_message_time):
    # 1. 间隔检查
    if (now - last_message_time) < config.interval_minutes:
        return False

    # 2. 时间窗口检查
    if is_rest_period(now, config.wake_time, config.sleep_time):
        if character_card.proactive_urgency < 0.8:
            return False

    # 3. 随机因子（±30%）
    offset = config.interval_minutes * config.random_factor * random.uniform(-1, 1)
    if (now - last_message_time) < (config.interval_minutes + offset):
        return False

    # 4. 起床/睡觉特殊消息
    if is_wake_time(now) and not wake_sent_today:
        return True, "wake_greeting"
    if is_sleep_time(now) and not sleep_sent_today:
        return True, "sleep_greeting"

    return True
```

#### 国产 ROM 保活方案（具体化）

### 7.3 高频用户压力测试

若用户每天 50 轮对话 + 10 次 LLM 提取 → 月成本可能接近 ¥15。

### 7.4 成本控制措施

1. Token 硬限制：单次请求上下文 ≤ 4000 token
2. LLM 提取节流：每 5 轮对话才触发一次
3. 主动消息日限额：每天最多 10 条

5. 本地成本计数器，APP 内显示"今日预估消耗"
6. **"省钱模式"**：降低 LLM 提取频率、关闭联网搜索、使用更小模型（若 DeepSeek 提供）
7. 用户可手动调整日预算上限

## 8. 风险评估全集

### 8.1 风险矩阵（R1-R11，两份报告合并）

| ID | 风险 | 概率 | 影响 | 等级 | 缓解措施 |
|----|------|------|------|------|----------|
| R1 | ChromaDB 在 Chaquopy 不可用 | 100% | 高 | **已落地** | DeepSeek Embedding API + 纯 Python 余弦相似度 |
| R2 | 国产 ROM 限制后台任务 | 80% | 高 | **高** | 前台服务保活 + 厂商适配引导（见 §5.3） |
| R3 | DeepSeek API 不稳定 | 30% | 高 | **高** | 3 次重试 + 降级回复 + 离线兜底 |
| R4 | 记忆提取质量不稳定 | 50% | 高 | **高** | 去重校验 + 矛盾检测 + 置信度字段 |
| R5 | Python→Android 性能落差 | 50% | 中 | **中** | P0 阶段做性能基准，限制 top_k |
| R6 | 主动消息时机不当 | 40% | 中 | **中** | 冷却期 + 用户活跃检测 + 全局开关 |
| R7 | API 成本失控 | 30% | 中 | **中** | Token 限制 + 日预算上限 + 省钱模式 |
| R8 | 上下文过长导致延迟 | 60% | 低 | **低** | 动态裁剪 + 摘要压缩 |
| R9 | API Key 泄露 | 15% | 中 | **低** | Android Keystore 加密存储 |
| R10 | Embedding API 向量维度变动 | 5% | 中 | **低** | 固定维度校验，变动时触发告警并重建索引 |
| **R11** | **WorkManager 多进程下 Python 全局状态丢失** | **60%** | **高** | **高** | **所有状态持久化到 SQLite，不依赖进程内全局变量** |

### 8.2 R11 详细说明（新增关键风险）

**问题**：WorkManager 可能运行在独立进程，而 Chaquopy 的 Python 环境在每个进程中独立初始化。因此 `app_context.py` 中的单例模式将失效。

**解决方案**：
- 所有状态（`last_message_time`、配置、对话历史）**必须持久化到 SQLite 或 SharedPreferences**
- 每次 WorkManager 触发 Python 调用时，从数据库重新加载状态
- 不依赖 Python 进程内的全局变量或单例

### 8.3 离线兜底方案（具体化）

当网络不可用或 API 失败时：
- 返回预设回复：`"我现在有点断线，稍后再聊好吗？"`
- 记录重试队列，网络恢复后自动重试
- 离线时主动消息功能暂停，避免生成无效通知

### 8.4 关键边界条件

- **空状态**：首次安装、无记忆、无对话历史
- **极限状态**：对话 500+ 轮、记忆 500 条、单条消息 5000 字
- **异常输入**：纯表情、Unicode 特殊字符、注入攻击
- **状态切换**：WiFi/4G 切换、息屏/亮屏、APP 后台/前台、手机重启
- **并发**：用户快速发送 + 主动消息同时到达
