## 二、第一个核心矛盾：Python 和 纯手机 APP 怎么共存？

这是整个项目最关键的技术决策。Python 不能直接编译成原生 Android APK（至少不能简单做到）。

### 方案对比

| 方案 | Python在哪跑 | 手机端是什么 | 优点 | 缺点 |
|------|-------------|-------------|------|------|
| **A: Termux 方案** | 手机 Termux 里 | Kotlin WebView / Compose 壳 | Python 代码完整复用；手机独立运行 | 用户需装 Termux；后台保活麻烦；不是"一个 APK 搞定" |
| **B: 服务器方案** | 你的 PC / 云服务器 | 纯 Android 客户端（Kotlin） | Python 环境舒适；手机端轻量；架构清晰 | 需要网络；需要一台常开的机器 |
| **C: Kivy 方案** | 手机内（Python→APK） | Kivy UI | 纯 Python 代码；一个 APK | UI 不像原生；后台服务不稳定；生态较小 |
| **D: Chaquopy 方案** | 手机内（嵌入 APK） | Kotlin Compose + Python 引擎 | 原生 UI + Python 逻辑 | 增加 APK 体积；包兼容性受限 |
| **E: PWA 方案** | 服务器 | 手机浏览器 / WebView 包装 | 跨平台；开发快；Python 后端 | 通知能力弱；不像原生 APP |

### 我的建议

> **推荐方案 B（服务器方案）**，理由：
> 1. 你已经有 PC，Python 环境现成
> 2. 架构最清晰：后端 Python 负责所有 AI 逻辑，前端 Android 负责展示和通知
> 3. 主动消息最可靠：服务器常开，定时任务稳定
> 4. 以后换本地 LLM 也方便（服务器上跑）
> 5. 手机端只是一个"聊天客户端"，可以做得非常轻量
>
> 如果你一定要手机完全独立运行，那方案 A（Termux）是次选。

## 三、主动消息的技术路线

> 你说"需要微信的原因就是因为可以主动发消息"——在独立 APP 里，这完全可以做到，而且不需要微信。

### 实现方式

```
服务器端（Python）：
  APScheduler 定时任务
  → 检查是否应该发消息（时间窗口、用户活跃度、今日已发次数）
  → 调用 DeepSeek 生成消息内容
  → 通过 Firebase Cloud Messaging (FCM) 推送到手机

手机端（Android）：
  收到 FCM 推送
  → 弹出本地通知（样式像微信消息：头像 + 昵称 + 内容预览）
  → 用户点击通知 → 打开 APP → 进入聊天界面
```

### 需要确认的问题

1. **FCM 在国内不稳定**，需要国内推送通道（如华为 HMS、小米 Push、OPPO Push 等），或者用统一推送联盟。你打算怎么处理推送？
2. 如果推送不可靠，是否可以接受**轮询方案**（APP 定期检查新消息）？
3. 主动消息的频率预期是什么？（比如：每 4 小时一次？每天早晚各一次？）

## 四、记忆系统的设计

你要求"像三次元真人的记忆"，这需要三层记忆架构：

```
┌─────────────────────────────────────┐
│         短期记忆（Short-term）         │
│  最近 N 轮对话，直接拼入上下文          │
│  容量：最近 50-100 条消息              │
│  作用：保证对话连贯性                  │
└──────────────┬──────────────────────┘
               │ 定期筛选、压缩
┌──────────────▼──────────────────────┐
│         长期记忆（Long-term）           │
│  从对话中提取的关键事实                  │
│  例如：用户喜欢吃什么、生日、重要事件      │
│  存储：结构化 JSON + 向量化用于检索       │
│  作用：跨会话的记忆保持                  │
└──────────────┬──────────────────────┘
               │ 关联分析
┌──────────────▼──────────────────────┐
│         关联记忆（Associative）         │
│  记忆之间的关联关系                     │
│  例如："喜欢猫" ← 关联 → "去过猫咖"     │
│  作用：让 AI 能"联想"到相关记忆          │
└─────────────────────────────────────┘
```

### 需要确认的问题

1. **长期记忆的提取方式**：是由 LLM 自动从对话中提取（异步），还是手动标注？自动提取会增加 API 调用成本。
2. **记忆遗忘策略**：真人的记忆会淡忘——是否需要模拟这个？（比如旧记忆权重随时间衰减）
3. **关联记忆**：这个是最复杂的部分，初期是否可以先做短期+长期，关联记忆放后续？

### 世界书

这是一个新概念。需要确认：
1. **世界书的范围**：是多个角色共享一个世界观（比如"都市日常"、"奇幻世界"），还是每个角色有独立的世界设定？
2. **世界书的内容**：包括地点、规则、其他 NPC、历史事件等？
3. **世界书如何影响对话**：AI 需要根据世界书来约束角色的行为和知识范围？

## 六、复杂度和开发周期预估

### 按推荐方案 B（服务器方案）估算

| 阶段 | 内容 | 复杂度 | 大约工时 |
|------|------|--------|----------|
| **P0: 架构搭建** | 项目结构、配置系统、日志、API 基础框架 | 中 | 2-3 天 |
| **P1: 核心对话** | 角色卡引擎、上下文管理、DeepSeek 调用 | 中 | 3-4 天 |
| **P2: 记忆系统** | 短期记忆、长期记忆提取、向量检索 | 高 | 4-5 天 |
| **P3: Android 客户端** | 聊天 UI、消息收发、通知 | 中 | 4-5 天 |
| **P4: 主动消息** | 定时引擎、话题生成、推送 | 中 | 3-4 天 |
| **P5: 世界书** | 世界书管理、上下文注入 | 中 | 2-3 天 |
| **P6: 打磨** | 测试、优化、文档 | 低 | 3-4 天 |

**总计约 21-28 天**（可对话聊天的基础版约 7-10 天）

### 复杂度说明

- **核心对话**：已有 v1 代码可参考，主要是重构和增强
- **记忆系统**：这是最复杂的模块，向量检索 + LLM 提取 + 关联分析
- **Android 客户端**：如果你不熟悉 Kotlin，这部分学习成本较高
- **主动消息**：逻辑不复杂，但推送通道需要折腾
