## 1. 项目概述

### 1.1 项目定位

一个**纯本地运行的 Android AI 角色扮演聊天 APP**，单 APK 安装即用。核心目标是让 AI 角色像真人一样与用户聊天，具备记忆、主动发起话题等能力。

### 1.2 核心约束

| 约束 | 说明 |
|------|------|
| 无服务器 | 不能依赖任何服务器，纯本地运行 |
| 单 APK | 一个安装包，安装即用 |
| 低成本 | 月 API 成本控制在 ¥5 以内 |
| 好架构 | 可维护、可扩展，不能是屎山 |

### 1.3 分工

- **AI（Trae）**：负责全部代码编写
- **用户**：负责需求定义和测试验收

## 2. 需求确认

### 2.1 三大核心体验

#### 聊天质量 — "像真人在微信上聊天"

- 多段短对话为主（模拟微信聊天风格）
- 偶有长对话
- 角色卡驱动，人设稳定不崩塌
- 对话风格自然，避免机械感

#### 记忆系统 — "像真人的记忆"

三层记忆架构：

```
短期记忆：最近 50 轮对话，保证对话连贯性
    ↓ 定期筛选
长期记忆：
  ├─ 普通记忆：规则提取 + LLM 提取，时间衰减
  └─ 强化记忆：用户/AI 标记，几乎不衰减，检索权重 ×2
    ↓ 关联分析
关联记忆：记忆之间的关联关系（初期简化）
```

#### 主动消息 — "AI 能主动找你聊天"

- 用户设定起床时间和睡觉时间
- 起床时间 → 第一次主动消息
- 睡觉时间 → 最后一次主动消息
- 休息时段（睡觉→起床）默认不发，除非角色卡强烈想发
- 活跃时段按用户设定的间隔随机发送
- 消息内容结合角色卡设定和近期对话

### 2.2 角色卡 + 世界书

**角色卡**：JSON 格式，定义角色性格、背景、对话风格、记忆关注点等。

**世界书**：JSON 格式，定义世界观。默认"三次元现实世界"，用户可自行添加（如赛博朋克、奇幻世界等）。

### 2.3 语音功能

- 不需要每句话都转语音
- AI 角色卡自行决定是否发送语音
- TTS 文字转语音功能待定，暂不实现

## 3. 技术方案

### 3.1 方案选择：Chaquopy

**Chaquopy** = Android 原生 APP 内嵌 Python 引擎。

```
┌──────────────────────────────────────┐
│            Android APK               │
│                                      │
│  Kotlin/Compose UI（原生界面）         │
│         ↓ 调用                        │
│  Chaquopy Python 引擎                 │
│    ├── chat_engine/  角色扮演         │
│    ├── memory/       记忆系统         │
│    ├── api_client/   API 调用         │
│    ├── proactive/    主动消息         │
│    └── world_book/   世界书           │
│         ↓                            │
│  Android 基础层（SQLite、WorkManager） │
└──────────────────────────────────────┘
```

**选择理由**：
- Python 写 AI 逻辑，代码可复用 v1 的 chat_engine 和 api_client
- Kotlin Compose 写 UI，原生体验
- 单 APK 安装，不需要 Termux 或服务器
- Chaquopy 免费开源（MIT 许可证）

**技术事实**：
- Chaquopy 在 APK 中嵌入 Python 3.10/3.11 运行时
- 纯 Python 包（如 `requests`）完全兼容
- 含 C 扩展的包（如 `chromadb`、`sentence-transformers`）**不可用**
- APK 体积增加约 40-50MB

### 3.2 Python 包兼容性清单

| 包名 | Chaquopy 兼容性 | 说明 |
|------|----------------|------|
| `requests` | **完全兼容** | 纯 Python，v1 核心依赖 |
| `sqlite3` | **内置支持** | Python 标准库 |
| `json` | **内置支持** | Python 标准库 |
| `python-dotenv` | **完全兼容** | 纯 Python |
| `numpy` | **Chaquopy 预编译支持** | 有 Android wheel，约 4.8MB |
| `scikit-learn` | **Chaquopy 预编译支持** | 有 Android wheel，约 8.4MB |
| `chromadb` | **不可用** | 依赖 onnxruntime、hnswlib 等 C 扩展 |
| `sentence-transformers` | **不可用** | 依赖 PyTorch，太重 |
| `faiss-cpu` | **不可用** | 无 Android wheel |

### 3.3 向量检索方案：DeepSeek Embedding API + 纯 Python 余弦相似度

由于 ChromaDB 不可用，采用以下方案：

```
存储：SQLite（JSON 字段存储向量）
嵌入：DeepSeek Embedding API 生成向量
检索：纯 Python 余弦相似度计算
```

- 5000 条记忆以内，检索耗时约 50-100ms
- 零额外依赖，不增加 APK 体积
- 如果未来记忆量超过 10000 条，可引入 numpy 加速

### 5.2 记忆系统

#### 短期记忆

- 最近 50 轮对话，`deque` 数据结构
- 每轮对话后自动更新
- 直接拼入上下文（不经过检索）

#### 长期记忆

**普通记忆**：
- 规则提取（免费）：匹配固定句式，如"我喜欢XX"、"我住在XX"
- LLM 提取（付费）：重要对话（>20 轮或用户标记）才触发
- 时间衰减：`score = importance × e^(-λ × days)`
- 衰减系数 λ=0.01（约 70 天后权重减半）

**强化记忆（Pinned）**：
- 触发条件：用户说"记住"、"别忘了"、点 📌 按钮，或 AI 角色卡认定重要
- 衰减系数 λ=0.001（极慢）
- 检索时权重 ×2.0

#### 关联记忆

- 初期简化：同一对话 session 内的记忆自动关联
- 后续升级：基于关键词 + 向量相似度自动关联

#### 记忆提取 Prompt

```
你是一个记忆提取助手。从以下对话中提取值得长期记住的事实。

【角色设定】
{character_name}：{character_personality}

【对话内容】
{conversation_text}

【提取规则】
1. 只提取对角色有意义的、值得长期记住的事实
2. 重点关注：用户的个人信息、偏好、重要事件、情感表达
3. 如果对话是日常寒暄，返回空列表

【强化记忆判断】
如果对话中出现以下情况，标记 is_pinned=true：
- 用户明确说"记住"、"别忘了"、"很重要"
- 角色根据自身设定，认为这条信息极其重要

请以 JSON 格式返回：
```json
{
    "memories": [
        {
            "content": "事实描述",
            "importance": 1-5,
            "is_pinned": true/false,
            "keywords": ["关键词1", "关键词2"]
        }
    ]
}
如果没有任何值得提取的记忆，返回 {"memories": []}
```

### 5.3 主动消息引擎

#### 配置

```json
{
  "proactive": {
    "wake_time": "08:00",
    "sleep_time": "23:00",
    "interval_minutes": 120,
    "random_factor": 0.3
  }
}
```

#### 决策逻辑

```python
def should_send_message(now, config, character_card, last_message_time):
    # 1. 间隔检查
    if (now - last_message_time) < config.interval_minutes:
        return False

    # 2. 时间窗口检查
    if is_rest_period(now, config.wake_time, config.sleep_time):
        if character_card.proactive_urgency < 0.8:
            return False

    # 3. 随机因子（±30%）
    offset = config.interval_minutes * config.random_factor * random.uniform(-1, 1)
    if (now - last_message_time) < (config.interval_minutes + offset):
        return False

    # 4. 起床/睡觉特殊消息
    if is_wake_time(now) and not wake_sent_today:
        return True, "wake_greeting"
    if is_sleep_time(now) and not sleep_sent_today:
        return True, "sleep_greeting"

    return True
```

#### 与 Android WorkManager 的配合

```
WorkManager (每30分钟触发)
  → ProactiveWorker (Kotlin)
    → 调用 Python proactive.engine.check_and_notify()
    → 如果 should_notify=True
      → NotificationManager 弹出本地通知
      → 用户点击 → 打开 APP 聊天界面
```

## 6. 开发路线图

### 优化后的路线（基于架构师审核建议）

| 阶段 | 内容 | 目标 | 关键验证点 |
|------|------|------|-----------|
| **P0: Python 原型** | 核心对话 + 记忆 + 主动消息，命令行运行 | 验证 AI 逻辑正确 | DeepSeek API 连通、角色扮演质量、记忆提取准确率 |
| **P1: 架构 + Chaquopy 验证** | 项目骨架 + 验证 Chaquopy 核心依赖 | 确认技术方案可行 | `requests`/`sqlite3` 在 Chaquopy 中正常 |
| **P2: 核心对话** | 角色卡引擎、上下文管理、API 调用 | 能聊天 | 10 轮对话人设不崩塌 |
| **P3: 记忆系统** | 短期 + 长期记忆（混合提取、向量检索） | 有记忆 | 记忆提取准确率 > 80% |
| **P4: Android 封装** | Chaquopy 集成 + Kotlin UI + APK 打包 | 手机上能用 | APK 安装运行，内存不泄露 |
| **P5: 主动消息** | 定时引擎 + WorkManager + 通知 | 会主动 | 休息时段 0 误触发 |
| **P6: 世界书** | 加载、管理、按需注入 | 世界设定 | 与角色卡联动正常 |
| **P7: 打磨** | 测试、优化、关联记忆 | 可发布 | 24h 稳定性测试 |

### 路线调整说明

相比初版方案的关键调整：
1. **P1 加入 Chaquopy 可行性验证**：在架构搭建同时验证核心依赖，避免后期翻车
2. **P4 Android 封装提前**：从 Python 命令行到 APK 的跨越是最大技术风险，尽早验证
3. **P5 主动消息后移**：主动消息依赖 WorkManager（Kotlin 侧），Android 封装完成后才能真测试
4. **关联记忆推迟到 P7**：复杂度远超预期，初期用简单策略替代

## 8. 风险评估与缓解

### 8.1 风险矩阵

| ID | 风险 | 概率 | 影响 | 等级 | 缓解措施 |
|----|------|------|------|------|----------|

| R3 | DeepSeek API 不稳定 | 30% | 高 | **高** | 3 次重试 + 降级回复 + 离线兜底 |
| R4 | 记忆提取质量不稳定 | 50% | 高 | **高** | 去重校验 + 矛盾检测 + 置信度字段 |
| R5 | Python→Android 性能落差 | 50% | 中 | **中** | P0 阶段做性能基准，限制向量检索 top_k |
| R6 | 主动消息时机不当 | 40% | 中 | **中** | 冷却期 + 用户活跃检测 + 全局开关 |
| R7 | API 成本失控 | 30% | 中 | **中** | Token 硬限制 + 日预算上限 + 本地计数器 |
| R8 | 上下文过长导致延迟 | 60% | 低 | **低** | 动态裁剪 + 摘要压缩 |
| R9 | API Key 泄露 | 15% | 中 | **低** | Android Keystore 加密存储 |

### 8.2 关键边界条件

- **空状态**：首次安装、无记忆、无对话历史
- **极限状态**：对话 500+ 轮、记忆 500 条、单条消息 5000 字
- **异常输入**：纯表情、Unicode 特殊字符、注入攻击
- **状态切换**：WiFi/4G 切换、息屏/亮屏、APP 后台/前台
- **并发**：用户快速发送 + 主动消息同时到达

## 9. 交叉审核意见汇总

### 9.1 架构师审核要点

**严重问题**：
1. ChromaDB 在 Chaquopy 中不可用 → 已确认替代方案（DeepSeek Embedding API）
2. 主动消息在国产 ROM 上的可靠性极低 → 需要前台服务保活
3. 关联记忆复杂度被严重低估 → 推迟到 P7

**建议改进**：
1. 开发路线调整：P1 加入 Chaquopy 验证，P4 Android 封装提前
2. 遗漏模块：错误处理与降级、对话导出与备份、数据迁移
3. 世界书改为按需检索注入（节省 token）

### 9.2 QA 测试审核要点

**高风险**：
1. Chaquopy + ChromaDB 兼容性 → 已确认不可用，已确定替代方案
2. DeepSeek API 不稳定 → 需要重试 + 降级 + 离线兜底
3. 记忆提取质量 → 需要去重 + 矛盾检测 + 置信度

**关键测试策略**：
1. P0 阶段：API 连通性、基础对话、记忆 CRUD 单元测试
2. P2-P4：记忆提取准确率、主动消息无误触发、Token 不超限
3. P5：APK 安装运行、内存不泄露、24h 稳定性

**UX 风险**：
1. 回复延迟 > 3 秒 → 需要流式输出 + "正在输入..."动画
2. 记忆"张冠李戴" → 相似度阈值调优（建议 0.75 起步）
3. 主动消息内容空洞 → 结合最近对话生成个性化问候

### 9.3 后端开发审核要点

**可行性确认**：
- 所有 Python 模块在 Chaquopy 下**完全可行**
- `requests` 纯 Python，完全兼容
- `numpy` 有 Chaquopy 预编译 Android wheel
- `chromadb` 不可用，已确定替代方案

**向量检索方案**：推荐 DeepSeek Embedding API + 纯 Python 余弦相似度
- 5000 条记忆以内，检索耗时约 50-100ms
- 零额外依赖，不增加 APK 体积

**性能瓶颈**：
- 上下文过长 → 动态裁剪 + 摘要压缩
- API 调用延迟 → 流式输出 + "正在输入..."动画
- 向量检索在大数据量下变慢 → 初期不优化，监控后再决定

**v1 代码复用**：
- 可直接复用：card_parser (95%)、deepseek (90%)、web_search (85%)、base (100%)
- 需要重构：role_player (40%)、context_manager (70%)、storage (30%)
- 完全废弃：wx_adapter/、api_server.py、templates/
