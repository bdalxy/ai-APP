## 三、核心流程详解

### 3.1 记忆存储流程 (remember)

```
对话结束
  │
  ▼
_orchestrator.remember(turn_id, user_msg, ai_reply)
  文件: orchestrator.py
  │
  ├─→ 1. 规则模式提取（零 API 开销）
  │     extractor.extract_rule_based()
  │     文件: extractor.py
  │     识别: 姓名、年龄、喜好、事件等 20+ 规则模式
  │
  ├─→ 2. LLM 模式提取（每5轮触发一次）
  │     extractor.extract_llm_based()
  │     文件: extractor.py
  │     调用 DeepSeek API 提炼深层记忆
  │
  ├─→ 3. 冲突检测
  │     memory_types.detect_conflict()
  │     文件: memory_types.py
  │     检测新旧记忆的语义冲突
  │
  ├─→ 4. 向量化存储
  │     vector_store.add()
  │     文件: vector_store.py
  │     ├─→ 调用 embed_cached() 获取向量
  │     ├─→ 存入 SQLite (含向量、类型、重要性、时间戳)
  │     ├─→ 更新倒排索引
  │     └─→ 记录变更日志
  │
  ├─→ 5. 缓存失效
  │     cache.on_memory_change()
  │     文件: memory_cache.py
  │
  └─→ 6. 自动备份检查（每100条或24小时）
       backup.auto_backup_check()
       文件: backup.py
```

### 3.2 记忆检索流程 (recall)

```
用户发送消息
  │
  ▼
_orchestrator.recall(query_text, top_k=10)
  │
  ├─→ 1. 查缓存
  │     cache.get_retrieval(query_text)
  │     文件: memory_cache.py
  │     命中 → 直接返回（跳过后续步骤）
  │
  ├─→ 2. 混合检索
  │     retriever.retrieve(query_text)
  │     文件: retriever.py
  │     │
  │     ├─→ 策略1: 向量余弦相似度（主策略）
  │     │    vector_store.search_by_embedding()
  │     │    文件: vector_store.py
  │     │
  │     ├─→ 策略2: 倒排索引（关键词匹配）
  │     │    vector_store.search_by_keyword()
  │     │
  │     ├─→ 策略3: BM25 文本相关性（降级）
  │     │    bm25.scorer.score()
  │     │    文件: bm25.py
  │     │
  │     └─→ 策略4: 全表扫描（最终降级）
  │
  ├─→ 3. 时间衰减加权
  │     decay.apply_time_decay()
  │     文件: decay.py
  │     越久远的记忆权重越低
  │
  ├─→ 4. 写入缓存
  │     cache.put_retrieval(query_text, results)
  │
  └─→ 5. 返回结果
```

### 3.3 上下文构建流程 (build_context)

```
对话前
  │
  ▼
_orchestrator.build_context(query_text, history, profile)
  文件: orchestrator.py → context_builder.py
  │
  ├─→ 1. Token 预算分配
  │     总预算: 800 tokens
  │     ├─→ 核心记忆: 50% (~400 tokens)
  │     ├─→ 扩展记忆: 30% (~240 tokens)
  │     └─→ 最近记忆: 20% (~160 tokens)
  │
  ├─→ 2. 三维排序
  │     相关性(0.5) + 重要性(0.3) + 时效性(0.2)
  │
  ├─→ 3. 分层注入
  │     ├─→ 核心层: 高相关 + 高重要
  │     ├─→ 扩展层: 中相关 + 情境性
  │     └─→ 最近层: 最近对话摘要
  │
  └─→ 4. 格式化为 System Prompt 片段
```

## 四、存储结构

### SQLite 表结构

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| `memories` | 主记忆表 | id, content, embedding(blob), category, importance, created_at, decay_at |
| `inverted_index` | 倒排索引 | word, memory_ids |
| `relations` | 记忆关系 | source_id, target_id, relation_type, confidence |
| `tags` | 标签 | name, color |
| `memory_tags` | 记忆-标签关联 | memory_id, tag_name |
| `changelog` | 变更日志 | memory_id, action, old_content, new_content, timestamp |
| `backups` | 备份元数据 | backup_id, type, created_at, size, checksum |

### 记忆类型系统

| 大类 | 子类 | 示例 |
|------|------|------|
| `episodic` (情节) | conversation, event, experience | "用户昨天聊了旅行" |
| `semantic` (语义) | fact, preference, knowledge | "用户喜欢草莓蛋糕" |
| `user_fact` (用户) | identity, relationship, habit | "用户是程序员" |

## 五、懒加载设计

MemoryOrchestrator 中 6 个扩展模块采用懒加载：

```python
# orchestrator.py 中的懒加载模式
@property
def _cache(self) -> MemoryCache:
    if self.__cache is None:
        self.__cache = MemoryCache()
    return self.__cache

@property
def _backup(self) -> MemoryBackup:
    if self.__backup is None:
        self.__backup = MemoryBackup(self.vector_store)
    return self.__backup
```

**好处**：不使用的功能不初始化，节省内存。

## 七、性能设计要点

| 机制 | 参数 | 说明 |
|------|------|------|
| LLM 提取节流 | 每5轮触发1次 | 减少 API 调用 |
| 规则提取 | 零 API 开销 | 处理寒暄对话 |
| Embedding 缓存 | LRU(500) | 避免重复向量化 |
| 检索缓存 | TTL(60s) | 重复查询直接返回 |
| 热点缓存 | LRU(200) | 最近访问记忆缓存 |
| 备份自动检查 | 100条或24h | 自动触发备份 |
| 懒加载 | 6个子模块 | 减少初始化开销 |

## 八、关键学习要点

1. **编排器模式**：`MemoryOrchestrator` 是唯一入口，所有功能通过它统一调度
2. **混合检索**：向量相似度 + 倒排索引 + BM25 + 全表扫描，多策略降级
3. **规则+LLM 双模式提取**：寒暄用规则（零开销），重要对话用 LLM
4. **SQLite 自建向量存储**：不依赖外部向量数据库，适合 Android 离线场景
5. **时间衰减**：旧记忆权重随时间降低，半衰期可配置
6. **懒加载**：按需初始化扩展模块，节省内存
7. **缓存分层**：Embedding 缓存 + 检索缓存 + 热点缓存，多层防护
