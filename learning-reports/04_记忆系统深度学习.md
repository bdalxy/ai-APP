## 0、ChatMemory 开源生态对比

B站评论区常提到的 "ChatMemory" 实际上指的是 AI 记忆领域的开源项目生态。以下是我们的记忆系统与业界主流方案的对比：

| 维度 | **Mem0** (55k+ Stars) | **Letta/MemGPT** (13k+ Stars) | **Zep** (2.5k+ Stars) | **我们的记忆系统** |
|------|----------------------|------------------------------|----------------------|-------------------|
| 记忆提取 | LLM 自动提取 | LLM 自我编辑 | 规则+LLM 提取 | 规则+LLM 双模式（节流） |
| 记忆分类 | 用户/会话/代理 | 核心/归档 | 时序分层 | **细粒度11类** (profile/preference/relationship/...) |
| 去重 | 语义相似度 | 无内置 | 向量去重 | 批次内+跨批次双阶段去重 |
| 检索 | 向量检索 | 向量检索 | 向量+图谱 | **5策略降级** (向量→图谱→BM25→倒排→全表) |
| 记忆衰减 | 访问频次 | 时间衰减 | 时序权重 | 半衰期+访问频次+**时间感知** |
| 知识图谱 | 无 | 无 | ✅ 时序KG | ✅ 自动构建+遍历 |
| 冲突检测 | 无 | 无 | 无 | ✅ 规则+语义冲突检测 |
| 加密存储 | 无 | 无 | 无 | ✅ Fernet AES + XOR 双模式 |
| 提问过滤 | 无 | 无 | 无 | ✅ **独创** 提问/请求识别 |
| 平台适配 | 云端 | 云端 | 云端 | **Android 离线** (SQLite) |

### 我们的特色优势（取百家之长）

1. **Mem0 的自动提取** → 我们的规则+LLM双模式，寒暄用规则省Token
2. **Letta 的分层记忆** → 我们的 archiver + summarizer 三层摘要链
3. **Zep 的时序图谱** → 我们的 knowledge_graph 自动构建+遍历
4. **独创能力**：
   - 提问/请求过滤（"能连续发几条消息吗？"不会被误记为"用户事实"）
   - 细粒度11类记忆分类（user_profile / user_preference / user_relationship / ...）
   - 5策略混合检索降级（向量→图谱→BM25→倒排→全表）
   - 加密存储（Fernet AES + XOR 双模式）
   - Android 离线运行（SQLite + Chaquopy Python）

## 三、核心流程详解

### 3.1 记忆存储流程 (remember)

```
对话结束
  │
  ▼
_orchestrator.remember(turn_id, user_msg, ai_reply)
  文件: orchestrator.py
  │
  ├─→ 1. 规则模式提取（零 API 开销）
  │     extractor.extract_rule_based()
  │     文件: extractor.py
  │     识别: 姓名、年龄、喜好、事件等 20+ 规则模式
  │
  ├─→ 2. LLM 模式提取（每5轮触发一次）
  │     extractor.extract_llm_based()
  │     文件: extractor.py
  │     调用 DeepSeek API 提炼深层记忆
  │
  ├─→ 3. 冲突检测
  │     memory_types.detect_conflict()
  │     文件: memory_types.py
  │     检测新旧记忆的语义冲突
  │
  ├─→ 4. 向量化存储
  │     vector_store.add()
  │     文件: vector_store.py
  │     ├─→ 调用 embed_cached() 获取向量
  │     ├─→ 存入 SQLite (含向量、类型、重要性、时间戳)
  │     ├─→ 更新倒排索引
  │     └─→ 记录变更日志
  │
  ├─→ 5. 知识图谱更新 (NEW)
  │     knowledge_graph.extract_from_memories()
  │     文件: knowledge_graph.py
  │     从本轮记忆中提取实体和关系
  │
  ├─→ 6. 缓存失效
  │     cache.on_memory_change()
  │     文件: memory_cache.py
  │
  └─→ 7. 自动备份检查（每100条或24小时）
       backup.auto_backup_check()
       文件: backup.py
```

### 3.2 记忆检索流程 (recall)

```
用户发送消息
  │
  ▼
_orchestrator.recall(query_text, top_k=10)
  │
  ├─→ 1. 查缓存
  │     cache.get_retrieval(query_text)
  │     文件: memory_cache.py
  │     命中 → 直接返回（跳过后续步骤）
  │
  ├─→ 2. 混合检索（多策略降级）
  │     retriever.retrieve(query_text)
  │     文件: retriever.py
  │     │
  │     ├─→ 策略1: 向量余弦相似度（主策略）
  │     │    vector_store.search_by_embedding()
  │     │    文件: vector_store.py
  │     │
  │     ├─→ 策略2: 图谱增强检索 (NEW)
  │     │    retriever.graph_enhanced_retrieve()
  │     │    向量检索 → 图谱1-2跳遍历 → 召回关联记忆
  │     │    文件: knowledge_graph.py
  │     │
  │     ├─→ 策略3: 倒排索引（关键词匹配）
  │     │    vector_store.search_by_keyword()
  │     │
  │     ├─→ 策略4: BM25 文本相关性（降级）
  │     │    bm25.scorer.score()
  │     │    文件: bm25.py
  │     │
  │     └─→ 策略5: 全表扫描（最终降级）
  │
  ├─→ 3. 时间感知增强 (NEW)
  │     decay.TemporalWeight.adjust()
  │     文件: decay.py
  │     根据当前时段/周末/工作日调整记忆权重
  │
  ├─→ 4. 写入缓存
  │     cache.put_retrieval(query_text, results)
  │
  └─→ 5. 返回结果
```

| 大类 | 子类 | 示例 | 重要性基线 |

| | `semantic_concept` | 用户理解的抽象概念 | 0.45 |
| `episodic` (事件记忆) | `episodic_event` | "用户昨天去了公园" | 0.45 |

| | `episodic_activity` | "用户每周三去健身房" | 0.40 |

## 五、懒加载设计

MemoryOrchestrator 中 8 个扩展模块采用懒加载：

```python
# orchestrator.py 中的懒加载模式
def _get_cache(self) -> "MemoryCache":
    if self._cache is None:
        from src.memory.memory_cache import MemoryCache
        self._cache = MemoryCache()
    return self._cache

def _get_knowledge_graph(self) -> "KnowledgeGraph":
    if self._knowledge_graph is None:
        from src.memory.knowledge_graph import KnowledgeGraph
        self._knowledge_graph = KnowledgeGraph(self.vector_store, self.client)
    return self._knowledge_graph

def _get_summarizer(self) -> "MemorySummarizer":
    if self._summarizer is None:
        from src.memory.summarizer import MemorySummarizer
        self._summarizer = MemorySummarizer(self.vector_store, self.client)
    return self._summarizer
```

**好处**：不使用的功能不初始化，节省内存。新增的 KnowledgeGraph 和 Summarizer 同样采用懒加载。

## 八、关键学习要点

1. **编排器模式**：`MemoryOrchestrator` 是唯一入口，所有功能通过它统一调度
2. **混合检索**：向量相似度 + 图谱增强 + 倒排索引 + BM25 + 全表扫描，5策略降级
3. **规则+LLM 双模式提取**：寒暄用规则（零开销），重要对话用 LLM
4. **SQLite 自建向量存储**：不依赖外部向量数据库，适合 Android 离线场景
5. **时间衰减 + 时间感知**：旧记忆权重随时间降低，同时根据时段/周末匹配度调整
6. **懒加载**：按需初始化扩展模块，节省内存（8个子模块）
7. **缓存分层**：Embedding 缓存 + 检索缓存 + 热点缓存，多层防护
8. **知识图谱**：自动提取实体+关系，支持图谱遍历和关联记忆召回
9. **记忆摘要链**：三层压缩（单轮→会话→主题），保持长期记忆简洁
10. **重要性动态评估**：结合访问频率、时效性、关系引用，定期重评估

### 9.1 知识图谱自动构建（knowledge_graph.py）

参考 Mem0 和 Zep 的图记忆架构，实现了从记忆文本中自动提取实体和关系：

```
记忆文本 → LLM 实体提取 → 实体存储 (kg_entities)
         → LLM 关系抽取 → 关系存储 (kg_relations)
         → 图谱遍历 (1-2跳) → 关联记忆召回
```

**实体类型**：PERSON / LOCATION / ORG / EVENT / CONCEPT / TIME
**关系类型**：has_property / located_in / part_of / works_at / knows / occurred_at / causes / related_to / same_as

**核心方法**：
- `extract_from_memories()`: 从记忆列表批量提取实体和关系
- `traverse(start_entity, max_hops=2)`: BFS 遍历图谱发现关联记忆
- `find_related_memories(memory_ids)`: 通过图谱关系发现额外关联记忆
- `query(entity_name)`: 查询实体详情及关联

### 9.2 记忆摘要链（summarizer.py）

参考 Letta/MemGPT 的递归记忆管理，实现三层摘要压缩：

```
原始记忆(20条) → 单轮摘要(1条) → 会话摘要(1条) → 主题摘要(1条)
  Layer 1          Layer 2          Layer 3
```

**触发条件**：
- 同类型记忆 > 20 条
- 距上次摘要 > 50 轮对话
- 手动调用触发

**摘要策略**：使用 LLM 生成简洁摘要，保留关键信息（人名、地点、事件、数字、日期），删除冗余，旧记忆标记为归档。

### 9.3 时间感知增强（decay.py）

在原有衰减算法基础上新增时间上下文感知：

- `get_temporal_context()`: 返回当前时段（早/午/晚/深夜）、周末/工作日
- `TemporalWeight.adjust()`: 根据记忆创建时间与当前时间匹配度调整权重
  - 周末产生的记忆在周末 +0.1 boost
  - 工作日产生的记忆在工作日 +0.1 boost
  - 同时段记忆 +0.05 boost

### 9.4 图谱增强检索（retriever.py）

新增 `graph_enhanced_retrieve()` 方法：

```
向量检索 top_k 条 → 图谱遍历（1-2跳）→ 召回关联记忆 → 合并去重排序
```

在 `multi_strategy_retrieve()` 中作为第2策略，嵌入到向量检索和BM25之间。

### 9.5 重要性动态评估（consolidator.py）

新增 `reassess_importance()` 方法，定期重新评估记忆重要性：

- 频繁访问（>5次）：+0.1
- 30天未访问：-0.1
- 被关系引用：+0.05/条（最多+0.1）
- 总调整范围：-0.2 ~ +0.2

在 `run_maintenance()` 中集成，与合并、衰减、清理一起执行。
